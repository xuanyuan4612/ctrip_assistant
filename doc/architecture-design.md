# 赫尔墨斯（Hermes）系统架构设计文档

---

## 一、架构概述

### 1.1 系统定位

赫尔墨斯（Hermes）是面向科沃斯集团的风险控制 AI 智能体系统，基于大语言模型（LLM）+ 检索增强生成（RAG）+ 多智能体协作（A2A）技术构建。系统覆盖风控业务全流程，集成**七大核心功能模块**，形成"**风险发现 → 调查分析 → 处置追责 → 整改闭环**"的完整风控链路。

**当前目标规模**：支撑集团 **10,000 名内部用户**并发使用，管理约 **10TB 文档数据**，处理海量多模态证据材料（音频/视频/图像/PDF），在保持低响应延迟（P95 < 500ms）的同时，满足等保二级合规要求。

### 1.2 核心架构能力

| 维度 | 能力描述 |
|------|----------|
| **工作流编排** | 基于 LangGraph 的有状态工作流引擎，支持条件路由、断点续跑、人工守门（Human-in-the-Loop） |
| **知识增强** | 基于 PGVector 的多分区向量检索（RAG），每个业务阶段挂载独立知识库索引；混合检索（向量 + Elasticsearch 全文） |
| **多智能体协作** | 通过 A2A（Agent-to-Agent）协议与龟宝（HR）、西塞罗（法务）、波特（财务）协同；基于 RabbitMQ 的消息总线保证可靠投递 |
| **外部系统集成** | 与风控系统、OA 系统（BPM）、MDM 主数据管理系统的双向数据同步；消息队列驱动 + 熔断器保护 |
| **异步优先架构** | 基于 RabbitMQ 消息队列的全异步处理架构，用户操作立即返回（< 100ms），长耗时任务（多模态处理、报告生成、外部同步、LLM 推理）后台执行（返回 task_id + WebSocket 进度推送）；支持重试、死信、优先级队列 |
| **多模态处理管道** | 独立处理管线：音频（Whisper ASR + 说话人分离）、图像（PaddleOCR + CLIP 分类）、视频（OpenCV 关键帧 + 场景分析）、文档（unstructured.io PDF/Office 解析）；GPU 加速处理 |
| **数据分层存储** | 三层热温冷数据分级：热数据（活跃案件 < 90天）存 PostgreSQL + pgvector；温数据（近期归档 90天-2年）存 MinIO 对象存储；冷数据（历史 > 2年）存 NAS/磁带归档；自动迁移策略 + 可搜索元数据索引 |
| **水平弹性伸缩** | 所有服务均无状态设计，基于 Kubernetes 自动扩缩容；API 层 3-20 实例（HPA）；Worker 按任务类型分 8 个独立池独立扩缩 |
| **CDN 与缓存** | Nginx 缓存层 + Redis Cluster 提供文档预览/缩略图 CDN 分发；热点查询缓存加速（5min TTL）；视频 HLS 自适应码率分段流 |
| **可观测性** | LangFuse + Prometheus + structlog + Jaeger 四层监控体系，覆盖 LLM 调用、API 性能、分布式追踪、业务指标。Grafana 统一仪表板 |
| **分层记忆** | L1 感知记忆 (即时上下文) → L2 会话记忆 (多轮对话) → L3 案例记忆 (案件全生命周期) → L4 组织记忆 (跨案件知识沉淀)。记忆从短到长逐层固化，支持跨案件学习与经验复用 |

## 二、设计原则

### 2.1 架构原则

| 原则 | 说明 | 落地方式 |
|------|------|----------|
| **关注点分离 (SoC)** | 业务模块独立部署，共享基础设施横向复用 | 7 个业务模块各自维护独立的 LangGraph 子图；共享认证、知识库、审计日志等横切服务 |
| **人机协同 (HITL)** | 每个工作流阶段碳基守门，AI 辅助决策、人工最终确认 | LangGraph `interrupt_before` 机制在每个阶段节点前挂起等待人工审批 |
| **有状态工作流** | 支持案件断点续跑、阶段回溯、人工介入后恢复 | Redis Cluster-based Checkpointer 持久化工作流状态，支持 LangGraph thread 级别恢复 |
| **知识驱动** | 各阶段决策基于结构化知识库而非仅依赖 LLM 参数记忆 | PGVector 多分区向量检索 + Elasticsearch 混合检索，每阶段注入对应知识库上下文到 LLM Prompt |
| **协议解耦** | 外部系统通过 Adapter 模式隔离，不直接耦合 | 风控系统、OA、MDM、A2A 均通过独立 Adapter 封装，支持协议变更时最小化影响面 |
| **安全内建 (Security by Design)** | 等保二级要求嵌入架构设计而非事后修补 | JWT + RBAC + AES-256-GCM 字段加密 + 全量审计日志 + 软删除 |
| **弹性设计** | 关键节点失败自动重试，不可恢复时降级至人工接管 | 每个 StageNode 定义 max_retries 和 timeout；RabbitMQ 消息队列保证异步可靠性 + 死信队列 |
| **异步优先 (Async-First)** | 所有重计算操作（多模态处理、报告生成、外部同步、LLM 推理）异步执行，用户 API 立即返回 task_id | 基于 RabbitMQ 的消息驱动架构，9 类独立 Worker Pool（audio/image/video/doc/report/sync/a2a/llm/kb）；用户通过 WebSocket 或轮询获取结果 |
| **数据分层 (Data Tiering)** | 按数据访问频率和生命周期分层存储，控制热存储成本 | 热层（活跃案件 < 90天）：PostgreSQL 主库 + pgvector；温层（近期归档 90天-2年）：MinIO 对象存储；冷层（历史 > 2年）：NAS/磁带归档；自动迁移策略与可搜索元数据索引 |
| **水平扩展 (Horizontal Scale)** | 所有服务无状态，独立水平扩展 | API 层：K8s HPA 自动扩缩 3-20 实例；Worker 层：按任务类型独立 Pod 池扩缩；数据库：主从读写分离 + 2 读副本 |
| **多模态管道 (Multimodal Pipeline)** | 每种媒体类型独立处理管线，充分利用 GPU 加速，处理结果统一索引 | 音频管道（Whisper + 说话人分离）；图像管道（PaddleOCR + CLIP 分类）；视频管道（OpenCV 关键帧 + 场景分析）；文档管道（unstructured.io PDF/Office 解析） |
| **渐进加载 (Progressive Loading)** | 大文件不阻塞用户交互，先预览后加载 | 文档缩略图预生成 + 分页流式加载；视频 HLS 分段传输；音频流式播放；文件 Range 分段下载 |
| **渐进交付** | 支持按模块分期实施，共享基础设施先行 | Phase 1 先交付基础骨架 + 廉洁监察模块 + 基础设施升级（K8s/消息队列/ES）；其余模块基于共享底座按需扩展 |

### 2.2 关键架构决策 (ADR)

| ADR # | 决策 | 理由 | 替代方案（已否决） |
|-------|------|------|-------------------|
| ADR-001 | 使用 LangGraph 而非自研工作流引擎 | 原生支持 Checkpointer、interrupt、条件路由、streaming；LangChain 生态集成 | 自研状态机（维护成本高）；Temporal（过重，不适合 LLM 工作流） |
| ADR-002 | PGVector 而非独立向量数据库 | 减少运维复杂度；与业务数据共库，支持事务一致性；pgvector 0.7+ HNSW 索引性能已满足规模需求（配合索引优化与分区） | Milvus（部署重）；Pinecone（SaaS 外部依赖） |
| ADR-003 | RabbitMQ + Celery 替换原有 Celery + Redis 直接架构 | 10K 并发需要可靠的消息持久化、集群 HA、消息优先级、死信队列（DLX）、联邦支持；RabbitMQ 提供企业级可靠性 | Redis 直接做队列（持久化弱，无原生优先级）；Kafka（日志语义，不适合任务队列场景） |
| ADR-004 | DeepSeek 为主 LLM，通义千问为备 | DeepSeek 中文能力优秀，API 兼容 OpenAI 格式，成本可控；通义千问作为降级备份 | GPT-4（成本高）；文心一言（API 兼容性差） |
| ADR-005 | 知识库离线本地管理，无飞书 API 依赖 | 等保要求数据不出内网；离线管理满足知识库低频更新场景 | 飞书 API 同步（外部网络依赖） |
| ADR-006 | Elasticsearch 作为全文搜索引擎 | 10TB 文档规模下，PostgreSQL 全文检索无法满足 P95 < 200ms 要求；ES 提供分布式倒排索引 + IK 中文分词 + 聚合分析 | PostgreSQL FTS（扩展性不足）；Solr（社区不如 ES 活跃） |
| ADR-007 | PaddleOCR 为中文 OCR 引擎（自部署） | 中文场景准确率 > Tesseract；支持表格识别、印章检测、版面分析；GPU 加速；纯内网部署，满足等保要求 | Tesseract（中文弱）；百度 OCR API（外部依赖） |
| ADR-008 | Whisper large-v3（自部署）为语音转文字引擎 | 开源、多语言支持、GPU 加速、可批量离线处理；1 小时音频 < 5 分钟处理（V100 GPU） | 阿里云语音识别（外部依赖、按量计费）；讯飞（API 限额） |
| ADR-009 | Kubernetes 替代 Docker Compose | 10K 用户规模需要自动扩缩容、滚动更新、健康检查、资源调度；Docker Compose 仅适合单机原型验证 | Docker Swarm（功能弱）；Nomad（生态小） |
| ADR-010 | Jaeger 分布式追踪 | 微服务化后需要跨服务调用链追踪，Jaeger 支持 OpenTelemetry、采样策略灵活（正常 10%，错误 100%） | Zipkin（功能类似但社区不如 Jaeger 活跃）；SkyWalking（偏 Java 生态） |

### 4.2 模块一：廉洁监察（反舞弊调查）✅ 已实现

**职责**：接收双轨案件来源（系统抓取 + 人工录入），通过 6 阶段 AI 辅助工作流完成从线索初判到报案协助的全流程反舞弊调查。

**工作流阶段**：

```
案件录入 → [4.1] 材料初判+分流 → [4.2] 调查方案 → [4.3] 多维分析+报告
                                        ↓
         [4.6] 报案协助 ← [4.5] 处罚执行 ← [4.4] 处置分流+追责
```

**状态定义** (`HermesState`)：包含案件基础信息（task_id, fraud_source, client）、工作流控制（current_stage, stage_history）、分流决策（should_investigate, is_hr_related）、各阶段 AI 输出（intake_report, case_conclusion, penalty_opinion）、碳基守门状态（pending_approval_stage, approval_result）、外部系统状态（risk_control_sync_status, a2a_task_ids）。

**外部交互**：
- 风控系统：字段映射同步（fraudSource ↔ reportedStaffName 等双向映射），WebSocket 状态推送
- A2A 龟宝：HR 案件转交、员工处罚跟踪回传
- A2A 西塞罗：民事纠纷推送、处罚协议审核
- A2A 波特：供应商扣款跟踪
- OA 系统：添可事业部处罚公告审批推送
- MDM：涉案供应商黑名单维护

### 4.3 模块二：风险监控（主动风险扫描）⏳ 规划中

**职责**：实现 7×24 小时无人值守自动风险监控，从规则生成到异常数据捕获、分析定性、自动推送的全链路主动扫描。

**工作流阶段**：

```
[6.1] 风险规则清单生成 → [6.2] 异常数据生成 → [6.3] 风险分析与主体合并
                                                   ↓
[6.6] 处置回流与指标迭代 ← [6.5] 风险结果推送 ← [6.4] 风险自动定性
```

**核心能力**：
- AI 自动生成风险场景 → 计算规则 → SQL 语句，测试环境验证后可入库
- 双重校验机制：AI 初核（剔除正常业务）+ 人工二次复核
- 按分析主体自动合并重复预警，统一口径
- 自动判定风险类型（合规风险/舞弊风险/商业秘密风险）、等级、处置建议
- 处置结果回流至知识库，持续优化规则准确率

**外部依赖**：业务数据库（SQL 查询）、企查查工商信息、舆情信息、外部法律法规

**多模态数据**：

|----------|------|----------|----------|--------|

### 4.4 模块三：内控评价（合规评价）⏳ 规划中

**职责**：覆盖 19 个业务循环的内控设计与执行有效性评价，自动生成审计方案、访谈问卷、测试底稿、缺陷清单和评价报告。

**工作流阶段**：

```

                                                              ↓
出具报告 ← 内控评价总体打分 ← 缺陷确认 ← 执行缺陷评估 ← 设计缺陷评估
                                  ↑                          ↑
                           资料需求清单 ←────────────────────┘
```

**特殊依赖**：
- 内控评价模块维护的"制度库"需向商业秘密模块提供制度比对能力
- 对接西塞罗 Agent 的岗位职责功能，用于访谈人员匹配

**多模态数据**：

| 数据类型 | 来源 | 处理方式 | 预估量/项目 | 存储层 |
|----------|------|----------|------------|--------|
| 音频 (访谈录音) | 内控访谈 | Whisper ASR → 文本 | 5-15 段 (500MB-3GB) | 热层→温层 |
| 文档 (制度文件) | 公司制度/流程文档 | 文档解析 → 制度库索引 | 50-200 份 (50-200MB) | 热层(活跃) |

### 4.5 模块四：专项审计（专项检查）⏳ 规划中

**职责**：基于审计目的和重点，自动生成标准化审计方案、访谈问卷、资料检查计划，执行检查作业并输出审计报告。

**工作流阶段**：

```
审计方案生成 → 访谈作业 → 检查作业 → 问题确认 → 出具审计报告
```

**核心能力**：
- 审计方案自动匹配：检索与审计目的/重点关联的历史方案，综合输出优化方案
- 抽样策略 AI 建议：根据业务量级和风险程度建议样本量和抽样方法
- 多数据源分析：SQL 直查数据中台 + API 获取外部数据 + 人工上传窗口
- 问题逐条下发确认，AI 分析反馈合理性

**多模态数据**：

| 数据类型 | 来源 | 处理方式 | 预估量/项目 | 存储层 |
|----------|------|----------|------------|--------|

| 文档 (审计资料) | 被审计单位提供 | 文档解析 + 索引 | 30-100 份 (30-100MB) | 热层→温层 |
| 图像 (检查证据) | 现场检查拍照 | OCR + 分类 | 20-50 张 (30-100MB) | 热层→温层 |

### 4.6 模块五：离任审计（离职审查）⏳ 规划中

**职责**：根据被审计人职责配置和任职时间，自动生成离任审计方案，识别个人问题和业务问题。

**工作流阶段**：

```
审计方案生成 → 访谈问卷 → 资料清单 → 问题清单生成 → 问题确认 → 出具报告
```

**特殊匹配逻辑**：
- 读取被审计人岗位职责 → 在业务审计方案库中检索强关联内容 → 在个人审计方案库中检索通用检查项
- 审计期间自动计算：1 年<本岗位≤5 年 → 向前计算三年；>5 年 → 向前计算一年
- 问题清单二分：个人问题（商业秘密泄露、个人报销、样机使用、关联公司）+ 业务问题（流程漏洞、制度缺陷、经济损失）
- 接收其他模块（如风险监控）传回的行为风险预警结果

**多模态数据**：

| 数据类型 | 来源 | 处理方式 | 预估量/人 | 存储层 |
|----------|------|----------|----------|--------|
| 文档 (个人档案) | HR 系统导出 | 文档解析 + 索引 | 5-20 份 (5-20MB) | 热层 |
| 图像 (报销凭证) | 财务系统截图 | PaddleOCR + 表格提取 | 10-30 张 (20-50MB) | 热层→温层 |
| 文档 (离职材料) | 离职交接单/审计底稿 | 文档解析 | 3-10 份 (3-10MB) | 热层→温层 |

### 4.7 模块六：商业秘密保护（保密管理）⏳ 规划中

**职责**：提供保密员定密预审建议和集团评审小组定密信息评审，辅助定密流程的专业性和完整性。

**三大功能**：

### 4.8 模块七：持续改善（整改跟踪）⏳ 规划中

**职责**：统一承接所有上游模块（廉洁监察、内控评价、专项审计、离任审计）的审计问题清单，实现从录入、下发、整改、复核到归档的全流程闭环跟踪。是六大执行模块的数据汇合与整改统一出口。

**核心子模块**：

```
┌────────────────────────────────────────────────────────────┐
│                     持续改善智能体                          │
├───────────┬───────────┬───────────┬───────────┬───────────┤
│ 问题录入   │ 任务下发   │ 整改跟踪   │ 复核闭环   │ 问题归档   │
```

### 5.3 消息队列与异步处理

| 组件 | 选型 | 版本 | 用途 | 选型理由 |
|------|------|------|------|----------|
| **消息队列** | RabbitMQ | 3.13+ | 核心消息总线，9 个业务队列 + DLX 死信队列 | 消息持久化、ACK 确认、优先级队列、延迟消息、镜像队列 HA、联邦 |
| **任务分发** | Celery | ≥5.4 | Worker 任务执行框架 | 成熟的任务队列（Kombu 连接 RabbitMQ），支持重试、优先级、任务链 |

### 6.3 HPA 扩缩容策略

```
  自动扩缩 (HPA - Horizontal Pod Autoscaler):
  API 服务:
  ├── 指标: CPU > 70% 持续 2min → 扩容 +2
  ├── 指标: 请求延迟 P95 > 500ms 持续 2min → 扩容 +2
  ├── 指标: CPU < 30% 持续 10min → 缩容 -1
  ├── 冷却: 扩容 1min / 缩容 5min
  └── Pod Disruption Budget: minAvailable=2

  Worker 服务 (通用):
  ├── 指标: RabbitMQ 队列深度 > 阈值 → 扩容 +1
  ├── 指标: 队列深度 < 阈值/2 持续 10min → 缩容 -1
  └── GPU Worker: GPU 资源稀缺，优先保留而非缩容

  定时扩缩 (CronHPA):
  ├── 工作日 8:00 → API 预热到 5 副本 (应对上班高峰)
  ├── 工作日 9:00-12:00 → API 保持 8-20 副本 (核心工作时段)
  ├── 工作日 20:00 → API 缩容到 3 副本 (非工作时间)
  ├── 周末/节假日 → API 保持 2 副本 (风险监控 7x24)
  └── 风险监控 Worker: 7x24 最少 1 副本，工作日峰值 3 副本
```

```
  网络分区 (VLAN):
  VLAN 10: DMZ/接入层    → Nginx x2 (仅开放 80/443)
  VLAN 20: 应用层        → K8s Worker x5 + GPU Worker x2 (不对外暴露端口)
  VLAN 30: 数据层        → PostgreSQL/Redis/MinIO/ES/RabbitMQ/NAS
  VLAN 40: 管理层        → K8s Master/Harbor/Prometheus/Grafana/Jaeger/堡垒机

  防火墙规则:
  ├── DMZ → 应用层: 仅 TCP 8000 (API) + TCP 5672 (AMQP管理)
  ├── 应用层 → 数据层: TCP 5432(PG), 6379(Redis), 9000(MinIO), 9200(ES), 5672(AMQP)
  ├── 管理层 → 所有: SSH(堡垒机), Prometheus metrics, K8s API
  └── 出网白名单: DeepSeek API, 通义千问 API, 企查查 API, 外部Agent回调
```

### 6.5 数据库集群配置

```
  PostgreSQL 16 集群 (Streaming Replication + PgBouncer):
  Primary (8C/32G/2TB) → WAL Streaming → Replica-1 (8C/32G/2TB)
                                        → Replica-2 (8C/32G/2TB)
  分区策略:
  ├── cases:      按 created_at RANGE 分区 (年)
  ├── audit_logs: 按 created_at RANGE 分区 (月)
  ├── case_evidences: 按 case_id HASH 分区
  └── kb_chunks:  按 kb_type LIST 分区 + 按年子分区

  索引策略:
  ├── B-tree: 高频查询字段 (case_id, user_id, status)
  ├── GIN: JSONB 字段, 全文检索字段
  ├── HNSW: pgvector 向量索引 (kb_chunks 表)
  └── 部分索引: 仅活跃状态的案件索引

  备份策略:
  ├── WAL 连续归档 (streaming) → MinIO
  ├── 每日全量备份 (pg_basebackup) → MinIO + NAS
  ├── PITR: 支持任意时间点恢复 (保留 30 天 WAL)
  └── 备份异地存储 (MinIO + NAS 双副本)
```

### 7.2 模块间数据流详表

| 上游模块 | 下游模块 | 数据类型 | 交互频率 | 传输方式 | 数据量级 |
|----------|----------|----------|----------|----------|----------|
| **风险监控 → 廉洁监察** | 异常推送（舞弊风险事件） | 案件线索 | 准实时（事件触发） | RabbitMQ 消息 + DB写入 | 少量(KB) |
| **风险监控 → 内控评价** | 合规异常推送 | 异常明细 | 准实时 | RabbitMQ 消息 | 少量(KB) |
| **风险监控 → 商业秘密** | 泄密风险预警 | 预警事件 | 准实时 | RabbitMQ 消息 | 少量(KB) |
| **廉洁监察 → 持续改善** | 问题清单（处罚、整改建议） | 结构化问题条目 | 阶段完成时 | RabbitMQ 消息 + DB写入 | 少量(KB) |
| **内控评价 → 持续改善** | 缺陷清单（设计缺陷+执行缺陷） | 结构化缺陷条目 | 报告出具时 | DB写入 + 消息通知 | 少量(KB) |
| **内控评价 → 商业秘密** | 制度库（共享依赖） | 制度文件 | 按需查询 | API查询 + 消息队列批量同步 | 大量(MB-GB) |
| **专项审计 → 持续改善** | 审计问题清单 | 结构化问题条目 | 报告出具时 | DB写入 + 消息通知 | 少量(KB) |
| **离任审计 → 持续改善** | 问题清单（个人+业务） | 结构化问题条目 | 报告出具时 | DB写入 + 消息通知 | 少量(KB) |
| **离任审计 ← 风险监控** | 行为风险预警结果 | 预警数据 | 审计执行前 | API 查询 | 少量(KB) |
| **全模块 ↔ 多模态处理层** | 文件上传 → 处理结果 | 音频/图像/视频/文档 | 文件上传时异步触发 | MinIO 事件 → RabbitMQ → Worker → ES+PGVector | 大量(MB-GB/文件) |

> **传输方式升级说明**：模块间数据交互从"直接 API 调用 + 数据库写入"升级为"**RabbitMQ 消息驱动 + 数据库最终一致性**"。大文件传输（如制度库同步）通过 MinIO 预签名 URL 或消息队列元数据交换，避免大载荷阻塞消息总线。

## 八、非功能性需求设计

### 8.1 安全性设计（等保二级 + 10K 规模增强）

| 安全域 | 需求项 | 实现方式 |
|--------|--------|----------|
| **身份鉴别** | 用户身份唯一标识与验证 | JWT 令牌机制，Access Token 8h/Refresh Token 7天；失败 5 次锁定 30 分钟；Token 黑名单 (Redis Cluster 全局同步) |
| **访问控制** | 基于角色的细粒度权限 | RBAC 三级角色（集团/科沃斯/添可）；行级数据过滤（按 client）；大文档场景支持字段级权限 |
| **安全审计** | 全量操作可追溯 | 所有操作记录 `audit_log` 表（操作人/IP/时间/内容/结果）；日志仅追加不可删除；ES 索引保留 1 年可搜索，归档保留 10 年 |
| **通信保密性** | 传输层加密 | HTTPS (TLS 1.2+)；内网服务间 mTLS；API 限流 (全局 1000/s + 用户 100/min) |
| **数据保密性** | 敏感字段加密存储 | AES-256-GCM 列级加密（姓名/电话/邮箱）；密钥通过 K8s Secret 注入；MinIO SSE-S3 文件传输加密 |
| **数据分级保护** | 10TB 数据分类管控 | 密级标记：公开/内部/秘密/机密四级；敏感文件访问独立审计；机密级下载需风控负责人审批 |
| **软件容错** | 关键节点失败处理 | StageNode 最大重试 3 次 → human_intervention → 告警+人工接管；RabbitMQ 自动重试 + DLX 死信队列 |
| **资源控制** | 防止资源滥用 | 混合限流（全局 1000/s + 用户 100/min + LLM 令牌桶 50/s）；文件上传 500MB；PgBouncer pool_size=500 |
| **数据备份** | 数据可恢复 | PG 每日全量 + WAL 连续归档 (MinIO + NAS 双副本)；Redis AOF+RDB；MinIO EC 12+4 冗余 |
| **剩余信息保护** | 登出时清除会话 | JWT 黑名单 (Redis Cluster 全局同步)；敏感字段不输出到日志 |

### 8.4 可用性设计

**全局可用性目标**：99.9%（内部系统，年停机 ≤ 8.76 小时）

| 场景 | 策略 | RTO | 降级行为 |
|------|------|-----|----------|
| **LLM 不可用** | 自动切换备用 LLM（通义千问）；两者均不可用 → human_intervention，任务保留队列 | 0s | 降级：案件进入人工接管模式 |
| **DB 主库不可用** | Patroni 自动故障转移；Replica 提升为 Primary | < 30s | 降级：写操作暂停(队列缓冲) |
| **DB 读副本不可用** | 读请求路由到 Primary 或其他 Replica | 0s | 降级：性能下降 (Primary 承载读写) |
| **Redis 部分不可用** | Cluster 自动故障转移 | < 30s | 降级：Checkpointer 降级 MemorySaver |
| **ES 不可用** | 全文搜索降级到 PostgreSQL FTS | 0s | 降级：搜索变慢 (P95 可能 > 1s) |
| **RabbitMQ 不可用** | 镜像队列自动故障转移 | < 10s | 降级：新任务无法提交 |
| **MinIO 部分节点** | EC 12+4 容忍 4 节点故障 | 0s | 降级：读取可用 (EC 恢复) |
| **外部系统不可用** | Adapter 熔断器 (5 失败 → OPEN 60s → HALF_OPEN)；> 5min → P2 告警 | 0s | 降级：同步操作延后 |
| **多模态管道不可用** | 文件正常上传，处理任务保留队列 | 0s | 降级：文件已上传待处理 |
| **工作流中断** | Redis Checkpointer 持久化；thread_id 精确恢复 | < 10s | 无降级 |
| **节点故障** | K8s liveness probe 自动重启不健康 Pod | < 30s | Pod 迁移到健康节点 |

### 8.6 业务连续性设计

| 场景 | RPO | RTO | 实现方式 |
|------|-----|-----|----------|
| **数据库主库故障** | < 5min | < 15min | Patroni 自动故障转移 + PgBouncer 切换 |
| **数据库集群全故障** | < 1min (WAL最后归档) | < 2h | 从 MinIO + NAS 备份 pgBackRest 完整恢复 |
| **Redis 全集群故障** | < 1min (AOF最后同步) | < 15min | AOF+RDB 持久化；Cluster 自动重建 |
| **MinIO 集群全故障** | < 1h (最后备份) | < 1h | EC 冗余 + NAS 定期备份 (rsync) |
| **ES 全集群故障** | < 1h | < 1h | 快照备份到 MinIO (每日) → 快照恢复 |
| **RabbitMQ 集群故障** | 0 (持久化消息) | < 15min | 镜像队列 + 磁盘持久化 |
| **应用 (API/Worker)** | 0 (无状态) | < 1min | K8s 多副本 + 健康检查 + 自动重启 + HPA |
| **完整站点灾难** | < 1h | < 4h | 异地备份 (MinIO + NAS 跨机房)；季度灾备演练 |

**灾备演练要求**：每季度一次；范围含 DB 恢复 + Redis 恢复 + ES 索引重建 (必做) + 完整站点切换 (年度选做)；每次输出 RPO/RTO 实际值 + 改进计划。

### 8.8 数据生命周期管理

| 阶段 | 触发条件 | 迁移动作 | 存储介质 | 访问 SLA |
|------|----------|----------|----------|----------|
| **热层 (Hot)** | 案件创建至关闭后 90 天 | — | PostgreSQL + pgvector | P95 < 50ms (索引) / < 200ms (复杂) |
| **热→温** | 案件关闭后 90 天 | 案件完整数据打包 → MinIO；PG 中保留元数据指针和摘要；原始大文件从 PG 删除 | MinIO 对象存储 | P95 < 200ms (元数据) / < 1s (文件下载) |
| **温→冷** | 案件关闭后 2 年 | 案件 + 证据 + 报告完整打包 → NAS/磁带；MinIO 中删除；ES 索引保留可搜索元数据；PG 仅保留案件号、归档路径、关键词 | NAS / 磁带 | 按需恢复：< 24h |
| **冷保留** | 冷层已存储 | 保留 10 年（法规要求） | NAS / 磁带 | 按需恢复 |
| **到期清理** | 超过 10 年 | 审批后物理销毁 | — | — |

**自动化机制**：
- 每日定时任务扫描案件状态：`status=closed AND closed_at < now() - 90days` → 热→温迁移
- 每周定时任务：`closed_at < now() - 2years` → 温→冷迁移
- 冷恢复流程：用户 API 请求 → 后台创建恢复任务 → 机器人从磁带提取 → 暂存 MinIO → 通知用户下载 (7 天有效期)
- 迁移日志完整记录到 audit_log

### 8.10 零停机数据库迁移策略

| 迁移类型 | 策略 | 示例 | 风险等级 |
|----------|------|------|----------|
| **新增表** | 直接执行，无影响 | `CREATE TABLE new_feature (...)` | 低 |
| **新增列 (可空)** | 直接执行；应用代码先兼容新列，再填充数据 | `ALTER TABLE ADD COLUMN x TEXT` | 低 |
| **新增列 (非空+默认值)** | 分三步：① 加可空列 + 默认值填充 ② 数据回填 ③ 改 NOT NULL | `ALTER TABLE ADD COLUMN status VARCHAR DEFAULT 'pending'` | 中 |
| **修改列类型** | 新增列 → 双写 → 回填 → 切换读取 → 删除旧列 | `VARCHAR(50)` → `VARCHAR(100)` | 高 |
| **删除列/表** | 先标记废弃 (应用代码不再读写) → 下一版本删除 | 软删除周期：至少 1 个版本 | 高 |
| **索引变更** | PostgreSQL `CONCURRENTLY` 模式 (不锁表) | `CREATE INDEX CONCURRENTLY idx_xxx ON t(c)` | 低 |
| **数据迁移** | Celery 异步任务分批处理 (1K/批, 间隔 100ms) | 历史数据重新分块 → Embedding | 中 |

**迁移执行流程**：
1. Alembic 生成迁移脚本 → Code Review (重点: 锁表风险、回滚方案)
2. Staging 环境验证 (使用生产数据量级模拟)
3. 生产维护窗口执行 (凌晨 2:00-4:00，或 CONCURRENTLY 随时执行)
4. 预检查: `SELECT pg_size_pretty(pg_database_size('hermes'))`, 磁盘剩余 > 30%
5. 迁移执行 + 监控 (长时间锁检测告警: > 10s)
6. 验证: 冒烟测试套件 (10 个核心 API)

### 8.13 多租户数据隔离方案

系统涉及三个法人实体（科沃斯、添可、集团），需在共享基础设施上实现数据隔离：

| 维度 | 隔离方式 | 实现 |
|------|----------|------|
| **数据库层** | 共享数据库 + 应用层隔离 (行级) | 所有表包含 `client` 字段 (`ecovacs` / `tineco` / `group`)；PG Row-Level Security (RLS) 可选 |
| **应用层** | RBAC 三层角色 + 查询自动注入 | API 中间件自动注入 `WHERE client = $current_user_role`；`group` 角色跳过过滤（全量可见） |
| **文件存储** | MinIO Bucket 按 client 分区 | `hermes-ecovacs` / `hermes-tineco` / `hermes-group` 三个独立 Bucket |
| **搜索索引** | ES 索引按 client 别名 | `hermes-docs-ecovacs` / `hermes-docs-tineco` / `hermes-docs-group`；搜索时自动路由 |
| **缓存** | Redis key 前缀隔离 | `ecovacs:cache:...` / `tineco:cache:...` / `group:cache:...` |
| **审计** | 审计日志记录 client 上下文 | 每条日志标记 `client` 字段，支持按租户独立审计 |

> 隔离原则：科沃斯用户不可查询添可数据，反之亦然。集团用户拥有全局视角。所有跨租户查询在应用层拦截，不依赖数据库层。

```
└─────────────────────────────────────────────────────────────────┘
```

#### 8.14.2 记忆流转与生命周期

```
用户操作 → L1 感知记忆 ──LLM推理──→ L2 会话记忆 (多轮对话)
                                        │
                        ┌───────────────┘ 守门通过
                        ↓
                  L3 案例记忆 (阶段产物 + 决策链)
                        │
            ┌───────────┼───────────┐
            ↓           ↓           ↓
       后续阶段引用   问题汇入     相似案件推荐
       (继续调查)   持续改善模块   (风险监控回流)
                        │
                        ↓ 案件闭环
                  ┌─────────────────┐
                  │   L3 → L4 固化   │
                  │                  │
                  │  调查报告 → 案例库│
                  │  处置经验 → 规则库│
                  │  缺陷模式 → 指标库│
                  └─────────────────┘
```

#### 8.14.3 记忆检索策略

| 记忆类型 | 检索方式 | 触发时机 | 技术实现 |
|----------|----------|----------|----------|
| **L4 组织记忆** | 向量语义检索 + 全文关键词 + 结构化过滤 | 每个工作流阶段启动时 | PGVector (cosine similarity) + ES (BM25) + SQL (元数据过滤)；混合检索 + Re-ranking |
| **L3 案例记忆** | 结构化查询 + 相似案例检索 | 调查方案生成 / 案件定性 / 追责建议时 | SQL 查询历史案件 (同类型/同部门/同金额区间)；PGVector 相似案例召回 |
| **L2 会话记忆** | LangGraph State 直接读取 | 单阶段内后续 LLM 调用 | `state["messages"]` 自动注入；超长对话自动摘要压缩 (LLM 生成 200 token 摘要) |
| **L1 感知记忆** | 实时注入 | 每次 LLM 调用 | System Prompt 模板 + 阶段知识注入 + RAG 上下文组装 |

#### 8.14.4 上下文窗口管理策略

针对 LLM 上下文窗口有限（DeepSeek 64K tokens）的问题：

| 策略 | 触发条件 | 处理方式 |
|------|----------|----------|
| **滑动窗口** | messages 总 token > 32K | 保留最近 16K tokens + System Prompt + 早期摘要 |
| **自动摘要** | 单阶段对话超过 10 轮 | LLM 生成 200 token 结构化摘要（关键事实/决策/待办），替换早期详细对话 |
| **分层注入** | 每阶段启动 | System Prompt (固定) → 阶段知识 (RAG top 5, ~2K tokens) → 历史摘要 (~500 tokens) → 用户输入 |
| **引用而非复制** | 引用大段文档内容 | 仅保留文档 ID + 相关片段引用，完整文档按需再次检索 |

#### 8.14.5 跨案件学习闭环

```
案件 A (已完成) ──→ 案例记忆 (L3)
                        │
            ┌───────────┼───────────┐
            ↓           ↓           ↓
       调查报告入库   处罚经验提取   缺陷模式识别
       (案例库更新)   (规则库更新)   (指标库更新)
            │           │           │
            └───────────┼───────────┘
                        ↓
              组织记忆 (L4) 更新
                        │
            ┌───────────┼───────────┐
            ↓           ↓           ↓
       案件 B         风险监控      内控评价
       (调查方案     (规则命中率    (缺陷识别
        自动引用      提升)         更精准)
        先例)
```

**固化机制**：
- **自动固化**：案件闭环后，调查报告 + 关键决策 + 处置结果自动入库（结构化特征提取 + Embedding 索引）
- **人工审核**：高风险案件（涉及刑事/重大金额）需人工审核后入库
- **衰减/淘汰**：过时案例（法律法规已变更）标记为"仅供参考"，权重降低
- **指标迭代**：风险监控模块的处置结果回流 → 误报规则优化 → 准确率持续提升（已体现在 风险监控 [6.6] 阶段）

### 8.16 SLO/SLI 定义与容量规划

#### SLO 指标定义

| 服务 | SLI | SLO | 测量窗口 | 告警阈值 |
|------|-----|-----|----------|----------|
| **API 可用性** | 成功请求数 / 总请求数 (2xx/3xx) | ≥ 99.9% | 30 天 | < 99.5% → P1 |
| **API 延迟** | P95 响应时间 (不含 LLM 调用) | < 500ms | 7 天 | > 1s → P1 |
| **LLM 可用性** | 成功调用数 / 总调用数 | ≥ 99.5% | 7 天 | < 98% → P1 (切换备用) |
| **LLM 延迟** | P95 端到端推理时间 | < 15s | 7 天 | > 20s → P2 |
| **知识库检索** | P95 检索延迟 | < 200ms | 7 天 | > 500ms → P2 |
| **多模态处理** | 任务成功率 (完成/提交) | ≥ 99% | 7 天 | < 95% → P2 |
| **消息队列** | 消息积压 (队列深度) | < 500 | 实时 | > 1000 → P3 |
| **数据迁移** | 热→温迁移成功率 | ≥ 99.9% | 30 天 | < 99% → P2 |

**错误预算**：API 99.9% = 月停机 ≤ 43 分钟。消耗 > 80% 时冻结生产变更，优先稳定性修复。

#### 容量规划触发条件

| 资源 | 扩容阈值 | 缩容阈值 | 最大容量 | 扩容前置时间 |
|------|----------|----------|----------|-------------|
| **API Pods** | CPU > 70% 持续 5min 或 内存 > 80% | CPU < 30% 持续 10min | 20 pods | 即时 (HPA) |
| **Worker Pods (各类)** | 队列深度 > 阈值的 80% 持续 10min | 队列深度 < 10% 持续 30min | 各 3-8 pods | 即时 (KEDA) |
| **PG 存储** | 使用率 > 70% | — | 2TB (SSD) | 提前 2 周规划扩容 |
| **PG 连接数** | PgBouncer 活跃连接 > 400 | — | pool_size=500 | 即时 (调整参数) |
| **MinIO 存储** | 使用率 > 70% (~14TB) | — | ~20TB (可用) | 提前 1 月加节点 |
| **ES 存储** | 使用率 > 75% | — | 6TB (3×2TB) | 提前 2 周加节点 |
| **Redis 内存** | 使用率 > 70% | — | 16GB/节点 | 提前 1 周加节点 |
| **NAS 冷存储** | 使用率 > 80% | — | 50TB | 提前 2 月采购扩容 |
