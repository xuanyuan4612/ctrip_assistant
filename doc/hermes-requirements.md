# 赫尔墨斯（Hermes）风险控制智能体 — 总体需求文档 v1.1

## 一、项目概述

### 1.1 项目背景

赫尔墨斯（Hermes）是面向科沃斯集团的风险控制 AI 智能体，旨在通过大语言模型（LLM）+ 检索增强生成（RAG）+ 多智能体协作（A2A）技术，覆盖风控业务全流程的智能化升级。系统集成了**八大核心功能模块**，形成从风险发现 → 调查分析 → 处置追责 → 整改闭环的完整风控链路。

### 1.2 业务痛点总览

| 痛点领域 | 问题描述 |
|----------|----------|
| **线索发现被动** | 案件来源单一依赖举报，缺乏主动异常识别能力；人工无法 7×24 小时覆盖海量数据 |
| **调查决策依赖经验** | 案件定性走向（刑事/民事/内部）缺乏标准化指引，高度依赖人工多方询问 |
| **取证与存证困难** | 海量证据抓取与核心证据存证留档效率低 |
| **追责闭环断裂** | 内部追责环节费时费力，难以保证连贯性及处罚到位 |
| **方案无统一知识沉淀** | 审计方案编制依靠个人经验，缺乏系统化知识复用 |
| **数据处理困难** | 数据量大且多元，传统人工查询整理效率低下 |
| **制度/流程监控不足** | 制度设计的有效性与流程执行的有效性缺乏高效监控与评价 |
| **定密评审效率低** | 跨事业部跨部门的定密信息评审依赖多领域专业知识，人工评审效率不足 |
| **整改跟踪成本高** | 审计问题整改进度需人工手动跟踪统计，难以确保整改闭环质量 |

### 1.3 建设目标

- **主动发现**：从"被动等线索"转变为"主动找异常"，实现 7×24 小时无人值守监控
- **智能引导**：在案件调查方向上提供 AI 引导，提升决策效率与准确性
- **高效固证**：提升海量证据抓取与核心证据存证留档的效能
- **闭环管控**：保障处罚落地，形成完整风控闭环
- **标准化审计**：实现审计方案自动生成、审计报告自动输出
- **全流程追踪**：审计问题整改统一承接、全程跟踪、闭环管控

#### 1.5.4 关键闭环链路

**端到端闭环示例**：
1. **主动发现 → 立案调查 → 处罚整改**：风险监控发现异常 → 推送廉洁监察立案 → 调查取证 → 追责处罚 → 问题汇入持续改善 → 整改闭环
2. **定期审计 → 缺陷发现 → 整改跟踪**：内控评价/专项审计/离任审计执行 → 发现问题清单 → 汇入持续改善 → 下发整改 → 复核归档
3. **定密管理 → 制度支撑 → 持续优化**：商业秘密定密 → 调用内控制度库 → 发现制度缺失 → 汇入持续改善 → 制度完善
4. **行为分析 → 风险预警 → 分流处置**：行为风险分析异常 → 推送离任审计/风险监控 → 问题清单 → 汇入持续改善 → 处置闭环

## 二、功能一：廉洁监察（反舞弊调查）

### 2.1 当前流程

当前廉洁监察与反舞弊工作为纯人工（碳基）处理方式，涵盖：接收举报 → 材料初判 → 调查方案制定 → 数据分析 → 访谈 → 案件结论 → 追责处罚 → 报案协助。

### 2.2 业务痛点

- 线索发现被动：案件来源长期单一依赖于举报线索
- 调查决策依赖经验：案件定性与走向缺乏标准化指引
- 取证与存证困难：有效举报信息识别、海量证据抓取效率低
- 追责闭环断裂：内部追责环节连贯性及处罚落实难以保证

### 2.3 建设目标

- 拓宽案件来源：从"被动等线索"转变为"主动找异常"
- 提供智能调查引导：在案件调查方向上给出 AI 引导
- 实现高效固证：提升海量证据抓取与核心证据存证留档效能
- 保障处罚落地：解决追责环节连贯性及处罚到位问题

### 2.4 权限说明

| 角色 | 权限范围 |
|------|----------|
| 集团 | 可查看所有举报案件 |
| 科沃斯 | 仅可查看涉及公司为科沃斯的案件 |
| 添可 | 仅可查看涉及公司为添可的案件 |

所有输出物均支持方案变更、划词调整两种功能，导出格式限定在 Word、Excel 等常用格式。

### 2.5 工作流阶段

```
案件录入（双轨来源）
    ↓
[4.1] 材料初判 + 分流决策
    ├── 不处理 → 闭环（推送风控系统）
    ├── 转交HR → A2A龟宝 → 闭环
    ├── 转交其他部门 → 任务中心 → 闭环
    └── 继续调查 ↓
[4.2] 调查方案生成
    ↓
[4.3] 多维分析 + 案件报告撰写
    ↓
[4.4] 处置分流 + 追责确定
    ├── 不涉及追责 → 闭环（推送风控系统）
    └── 涉及追责 ↓
         ├── 刑事立案 → 撰写报案书
         ├── 民事纠纷 → 推送西塞罗
         └── 内部处理 → 追责意见
    ↓
[4.5] 处罚执行 + 跟踪
    ↓
[4.6] 报案后续协助
    ↓
    闭环
```

#### 2.5.1 阶段一：案件来源与初筛（Intake & Initial Screening）

**功能描述**：
- **双轨案件来源**：
  - a. 系统抓取：智能体从风控系统自动抓取举报案件信息
  - b. 人工录入：风控人员手动新增案件信息
- **多模态预处理**：针对语音类线索，具备语音转文字功能
- **分流决策**：出具材料初判报告，明确是否开展调查的结论及理由，以及处置方式（内部/刑事/民事）
- **分流处理**：包含是否处置、是否转交、是否归属 HR 三个节点；碳基守门后执行分流

**知识库内容**：公司组织架构、人员名单、员工岗位职责、客户清单、供应商清单、内部管理制度、外部法律法规

**输入输出物**：

| 环节 | 输入 | 输出 |
|------|------|------|
| 语音转文字 | 音频文件 | Word 文档 |
| 材料初判 | 字段 + 附件解析 + 语音转文字文档 | 材料初判报告 + 调查对象类型（员工/供应商/经销商） |
| 是否处理 | AI 初判结论 | 是/否（碳基守门选项） |
| Output1 | 是否处理=否 | JSON 文件（与风控系统交互闭环） |
| 是否转交 | 是否处理=是 | 是/否（碳基守门选项） |
| 是否归属HR | 是否转交=是 | 是/否（碳基守门选项），A2A龟宝或辛顿平台任务中心 |
| Output2 | 转交完成 | JSON 文件（与风控系统交互闭环） |

**任务ID生成规则**：来源缩写 + 年月日 + 顺序序号（公众号=GZ、手动=SD、邮箱=YX、智能体=ZN、电话=DH），如 GZ2025121102

#### 2.5.2 阶段二：智能调查方案筹备（Investigation Planning）

**功能描述**：
- **智能信息提取与匹配**：自动提取材料初判中的关键字段，进行关键信息匹配，提示后续碳基可能需要找哪些人开展访谈、去哪些系统抓取数据、过往类似案件调查方式
- **调查方案生成**：基于知识库，自动生成案件初步走向预判及调查方案

**知识库内容**：外部类似案件相关法条、公司各业务系统、公司智能体及对应功能/数据、过往舞弊案件及处理方案

**输入输出物**：

| 环节 | 输入 | 输出 |
|------|------|------|
| 形成案件初步走向及调查方案 | 初判报告 + 附件 + 语音转文字文件 | 调查方案模板.xlsx |

#### 2.5.3 阶段三：多样化分析及案件报告撰写（Diverse Analysis & Case Report Finalization）

**功能描述**：

- **多维数据碰撞**：智能体告知碳基需要获取哪些系统数据验证舞弊真实性，碳基导入数据后通过 SQL 进行初步筛选，智能体进行数据比对与深度分析
- **访谈提纲生成**：依据调查方案，自动输出拟访谈人员及初步访谈提纲
- **访谈记录沉淀**：线下访谈完成后，利用语音转文字功能提取关键信息
- **案件结论**：碳基上传现场走访记录，智能体结合数据分析结果、现场走访记录、访谈记录三方面全面分析，出具初步案件结论
- **智能报告撰写**：基于前期证据链和调查结论，配合知识库自动生成调查报告初稿

**知识库内容**：过往舞弊案件调查报告、报告模板及格式具体要求

**输入输出物**：

| 环节 | 输入 | 输出 |
|------|------|------|
| 调用数据中台数据分析 | 调查方案中 AI 判定的数据需求 | 数据中台数据分析报告 |
| 其他智能体数据分析 | 调查方案中 AI 判定的智能体分析报告需求 | 智能体数据分析报告 |
| 进入业务源系统取证 | 人工上传多文件 | 原始数据分析结果报告 |
| 生成访谈提纲 | 碳基确认的调查方案中的访谈对象 | 所有访谈人员的访谈提纲 |
| 现场走访 | 人工撰写的走访任务调查发现、照片、视频、录音、收集的文件 | AI 针对现场走访资料的分析报告 |
| 语音转文字 | 走访音频文件、访谈录音 | 语音转文字记录文件 + 访谈纪要 |
| 形成案件结论 | 中台数据分析报告 + 智能体分析报告 + 原始数据分析报告 + 现场走访分析报告 | 多维度总结案件定性结论的报告 |
| 撰写廉洁监察报告 | 案件结论 | 标准制式廉洁监察项目报告 + 所有相关文件作为附件下载 |

#### 2.5.4 阶段四：处置分流及处罚确定（Decision & Penalty Determination）

**功能描述**：

- **处置分流决策**：
  - 若不涉及追责处罚 → 自动状态闭环，于辛顿平台及风控系统内关闭举报事件
  - 若涉及追责处罚 → 依据定性自主判断是否存在司法及纠纷路线
    - **外部司法路线**：判定涉及刑事立案 → 撰写报案书 → 碳基守门后移交司法程序
    - **民事纠纷路线**：判定涉及民事纠纷 → 推送西塞罗智能体任务中心 → 法务部门处理
    - **内部处理路线**：形成内部追责意见 → AI 参照制度要求生成规范化处置方案
- **内部追责意见定稿**：AI 出具规范化追责意见后，风控人员与各高级管理层确定最终追责方案

**知识库内容**：过往舞弊案件调查报告、报告模板及格式要求、公司制度文件、追责审批流程、组织架构及分权

**输入输出物**：

| 环节 | 输入 | 输出 |
|------|------|------|
| 是否涉及追责处罚 | 案件结论（涵盖追责处罚字段） | 是/否（用于后续判断条件） |
| OUTPUT3 | 是否涉及追责=否 | JSON 文件（与风控系统交互闭环） |
| 未来处置走向分析报告 | 是否涉及追责=是 + 案件结论 | 专项案件法律路径走向及深度研判报告 |
| 推送西塞罗任务中心 | 选择民事纠纷的结果 | 交互文件（案件信息 + 证据清单） |
| 撰写报案书 | 选择刑事立案的结果 | AI 生成的案件报案书 |
| 内部追责初步意见 | 案件结论 | 内部追责初步意见 + 处罚涉及人员清单（员工/经销商/供应商） |

#### 2.5.5 阶段五：处罚执行及跟踪（Penalty Enforcement & Tracking）

**功能描述**：

- **处罚公告撰写**：智能体结合知识库内公告模板自动撰写处罚公告
  - 涉及添可：自动依照 OA 系统接入需求推送 JSON 文件及处罚公告至 OA 审批
  - 涉及科沃斯/集团：碳基手工上传最终处罚公告，推送至风控系统闭环
- **赔偿协议撰写**：碳基选择需生成的处罚相关协议，生成后推送西塞罗任务中心，法务部门审核
- **黑名单维护**：智能体自动将涉事供应商导入 MDM 黑名单库
- **供应商扣款跟踪**：推送涉及扣款任务到波特任务中心，财务监控扣款情况
- **员工扣款跟踪**：通过 A2A 对接龟宝，由龟宝承接员工处罚落实与跟踪

**知识库内容**：黑名单管理制度、赔偿协议模板、处罚公告模板、人员架构

**输入输出物**：

| 环节 | 输入 | 输出 |
|------|------|------|
| 撰写处罚公告 | 追责初步意见 + 处罚公告模板 | 固定版式处罚公告 |
| OA 交互 | 案件涉及实体=添可 | JSON 交互文件（推送 OA 审批） |
| 手工上传最终处罚公告 | 案件涉及实体=集团/科沃斯 + 用户输入 | JSON 文件（推送风控系统闭环） |
| 撰写协议 | 碳基选择待生成的人员及文件类型 | 特定人员协议文件 |
| 推送西塞罗（协议审核） | 碳基守门后的协议文件 | 法务部门审核过的确定版协议文稿 |
| 维护黑名单 | 碳基点击是否进入黑名单 | MDM 交互文件（涉案供应商基本信息） |
| 推送波特任务中心 | 供应商扣款信息 | 财务部门定期反馈扣款情况 |

## 三、功能二：风险监控（主动风险扫描）

### 3.1 当前流程

现有风险监控依赖人工识别与处置，存在指标覆盖不足、响应滞后等问题。

### 3.2 业务痛点

- **人工识别效率低，响应滞后**：无标准化指标的风险场景依赖人工判断，受经验、精力、工作时间限制，易漏判、误判，无法 7×24 小时覆盖
- **指标覆盖不足，监控范围有限**：现有指标依赖人工提炼与历史经验，数量少、覆盖窄，存在大量管控盲区

### 3.3 建设目标

- 实现 7×24 小时无人值守自动监控，实时捕捉异常
- 自动完成异常数据分析、原因定位、影响评估
- 自动完成风险等级、风险类型判定
- 自动输出日/周/月标准化风险报告
- 自动推送至相关处置模块并形成处置闭环

### 3.4 工作流阶段

```
[6.1] 风险规则清单生成（AI + 人工审核）
    ↓
[6.2] 风险异常数据生成（AI 跑数 + AI 初核 + 人工复核）
    ↓
[6.3] 风险分析与主体合并（AI 自动聚合）
    ↓
[6.4] 风险自动定性（AI 判定 + 人工可修正）
    ↓
[6.5] 风险结果自动推送
    ↓
[6.6] 处置结果回流与指标迭代（闭环核心）
```

### 3.5 知识库与数据需求

| 类别 | 内容 |
|------|------|
| **内部数据** | 业务数据库字段及字段含义、历史审计项目、历史风险案例、处置结果、组织架构（事业部/渠道/业态/部门/岗位/人员信息） |
| **外部数据** | 企查查工商信息、舆情信息、最新法律法规、历史案件判例 |
| **知识库** | 风险清单知识库 |

### 3.6 各阶段详细说明

#### 3.6.1 风险规则清单生成（AI + 人工审核）

- **功能**：AI 自动生成风险场景、计算规则、SQL，人工审核入库
- **输入**：风险规则（事业部/渠道/业态/部门/岗位/一/二/三级场景）、数据库字段及含义、历史案件判例、人工自定义场景（可选）
- **逻辑**：
  1. 选择模式：清单上传 / 人工录入
  2. AI 生成三级场景 → 计算规则 → SQL 语句
  3. AI 在测试环境验证 SQL 可执行
  4. 人工审核通过 → 入库；不通过 → 修正/作废
  5. 形成风险清单知识库
- **输出**：风险清单知识库（标准字段：事业部、渠道、业态、部门、岗位、人员信息、业务循环、一级场景、二级场景、三级场景、SQL 语句、风险等级、阈值、监控频率、监控事业部、是否调用外部数据）

#### 3.6.2 风险异常数据生成（AI 跑数 + AI 初核 + 人工复核）

- **功能**：按规则自动跑出异常数据，双重校验
- **输入**：风险清单知识库、业务数据、外部数据
- **逻辑**：
  1. AI 按 SQL 与监控频率自动执行，生成异常明细
  2. AI 初核：剔除正常业务、标注异常说明
  3. 人工二次复核：通过 → 进入分析；不通过 → 调整规则/指标作废/回流优化
- **输出**：风险预警异常数据明细及透视表、风险预警异常数据分析报告

#### 3.6.3 风险分析与主体合并（AI 自动聚合）

- **功能**：按分析主体合并重复预警，统一口径
- **输入**：异常风险明细、异常风险明细透视表
- **逻辑**：
  1. AI 识别分析主体（联系方式、联系人、地址等）
  2. 同一主体多条异常自动合并，标记涉及指标
  3. 识别风险行为、风险业务、影响范围、涉及金额
- **输出**：按分析主体合并后的风险透视表、单主体风险分析报告

#### 3.6.4 风险自动定性（AI 判定 + 人工可修正）

- **功能**：自动判定风险类型、等级、影响、建议
- **输入**：单主体风险分析报告、历史审计案例知识库
- **逻辑**：AI 自动输出风险类型（合规风险/舞弊风险/商业秘密风险等）、风险等级、严重性、广泛性、影响程度（岗位/金额/业务范围/发生频次综合评估）、处置建议
- **输出**：风险定性报告及处理建议

#### 3.6.5 风险结果自动推送

- **功能**：自动分发至对应处置模块
- **推送对象**：业务部门智能体、廉洁监察模块、内控内审模块、商业秘密模块
- **输出**：推送记录

#### 3.6.6 处置结果回流与指标迭代（闭环核心）

- **功能**：接收处置结果，优化规则，避免误报
- **输入**：各模块处置结果、整改情况
- **逻辑**：
  1. 记录状态：已处置 / 整改中 / 无需处置 / 误报
  2. 误报/失效指标 → 人工确认 → 优化 SQL/阈值
  3. 更新风险清单知识库，持续提升准确率
- **输出**：风险处置闭环台账、指标优化迭代记录

## 四、功能三：内控评价

### 4.1 当前流程

风控系统立项 → 制定项目方案 → 发送审计通知 → 访谈阶段 → 测试阶段（穿行测试 & 有效性测试）→ 输出各循环内控缺陷（设计缺陷 + 执行缺陷）→ 按照评分标准打分 → 输出内控评价报告 → 各业务部门跟进整改

共覆盖 **19 个业务循环**，各有设计缺陷和执行缺陷两张底表作为底稿。

### 4.2 业务痛点

1. 制度设计的有效性缺乏高效的监控及评价
2. 流程执行的有效性缺乏高效的监控及评价
3. 全业务流程的内控评价项目花费大量时间，但不一定能产出重大问题，成果容易被管理层忽视

### 4.3 建设目标

1. 满足外部监管、审查需求
2. 及时从不同维度（主体、循环、岗位）评价设计和执行缺陷
3. 内控评价作为风控常规作业，大幅缩减业务时间
4. 提高缺陷整改沟通效率

### 4.4 知识库与数据需求

| 序号 | 内容 |
|------|------|
| 1 | 公司制度库 |
| 2 | 公司内控评价标准矩阵 |
| 3 | 《企业内部控制基本规范》《企业内部控制应用指引》《企业内部控制评价指引》 |
| 4 | 公司历史内控缺陷清单（含设计缺陷和执行缺陷） |
| 5 | 公司组织架构、花名册、岗位职责 |
| 6 | 19 个业务循环所涉及的公司各系统数据（业务数据 + 流程数据） |
| 7 | 方案、访谈清单、报告模板 |
| 8 | 历史改善建议 |

### 4.5 智能体输出成果

不同维度（主体、循环、业务）的内控评价报告（含具体缺陷、缺陷类别、整改建议、整改责任部门等）

### 4.6 工作流详细步骤

```
[7.1] 立项
    ↓
[7.2] 审计方案生成（5部分）
    ├── 项目基本信息
    ├── 评价依据
    ├── 审计范围
    ├── 审计实施细则
    └── 缺陷认定标准
    ↓
[7.3] 生成访谈人员清单及访谈计划
    ↓
[7.4] 生成访谈问卷
    ↓
[7.5] 根据访谈问卷完善风控矩阵
    ↓
[7.6] 拆分风控矩阵（设计测试底稿 + 执行测试底稿）
    ↓
[7.7] 评估内控设计缺陷（含打分）
    ↓
[7.8] 生成资料需求清单
    ↓
[7.9] 评估内控执行缺陷（含打分）
    ↓
[7.10] 内控缺陷输出及确认
    ├── 业务初步确认
    ├── 输出缺陷汇总及沟通确认表
    └── 与业务负责人确认
    ↓
[7.11] 内控评价总体打分
    ↓
[7.12] 出具内控评价报告
```

#### 7.2 审计方案生成

| 组成部分 | 内容说明 |
|----------|----------|
| 项目基本信息 | 项目名称、审计目的、被审计单位、审计期间、项目组长、项目组员 |
| 评价依据 | 标准内容，调用知识库内的方案模板 |
| 审计范围 | 根据风控系统同步的"标准控制活动选择"底表，获取业务循环和一级业务流程 |
| 审计实施细则 | 各阶段安排 + 各业务循环审计实施细则（含审计目标、关键控制点识别、审计测试程序、时间安排、人员分工） |
| 缺陷认定标准 | 标准内容，调用知识库内的方案模板 |

#### 7.7 评估内控设计缺陷

根据设计测试底稿（含制度编号、名称），按序阅遍相关制度文档，结合历史设计缺陷、设计缺陷类型、流程设置情况，根据评分标准打分，输出设计缺陷类型、缺陷描述。

#### 7.9 评估内控执行缺陷

根据系统对接数据/手动上传数据及流程测试数据，结合历史执行缺陷、执行缺陷类型、评分标准，输出缺陷类型、缺陷描述。

#### 7.10-7.11 缺陷确认与打分

- 根据打分规则对经确认的缺陷清单进行打分
- 设计缺陷评分标准和执行缺陷评分标准分别制定
- 将设计缺陷和执行缺陷清单与业务部门初步沟通确认
- 按业务循环、缺陷类型、一阶管理部门维度进行汇总统计
- 对各控制事项进行打分，按事业部、循环维度计算总分

#### 7.12 出具内控评价报告

报告结构：
- 一、项目概述（评价目的、评价范围、评价方式、评价依据）
- 二、内部控制整体结论（AI 根据缺陷清单出具整体结论）
- 三、内部控制运行情况（按公司主体汇总得分、缺陷类型、缺陷认定、缺陷数量）
- 四、重要缺陷列示
- 五、持续改善推进（整改措施确认截止时间为报告出具后 2 个月）

## 五、功能四：专项审计

### 5.1 当前流程

- 数据量大且多元造成数据处理困难

### 5.3 知识库与数据需求

| 内容 |
|------|
| 历史项目审计发现清单 |
| 历史审计方案 |
| 公司组织架构、花名册、岗位职责 |
| 各个业务循环所涉及的公司各系统数据（业务数据 + 流程数据 + 行为数据） |
| 访谈问卷模板 |
| 历史改善建议 |

### 5.4 智能体输出成果

- 标准化审计方案（含范围、目标、方法、程序）
- 审计工作底稿（半自动填充）
- 审计报告（草稿及正式版）
- 审计问题清单

### 5.5 工作流详细步骤

```
（一）审计方案生成
    ↓
（二）访谈作业
    ↓
（三）检查作业
    ↓
（四）问题确认
    ↓
（五）出具审计报告
```

#### （一）审计方案生成

- **功能**：AI 根据审计目的、审计重点，结合历史方案库、历史审计记录，自动生成包含审计范围、调研方法、样本策略、时间计划、人员规划的详细审计方案
- **输入**：
  - (1) 审计目的、审计重点、审计区间、被审计单位、项目组长、项目组员（来自风控系统立项）
  - (2) 审计方案库、审计记录库（来自知识库）
- **匹配逻辑**：
  - 审计范围及审计程序：检索与审计目的/重点关联的历史方案，综合输出优化方案
  - 抽样方案：AI 根据业务量级和风险程度建议样本量和抽样方法
- **输出**：《专项审计实施方案》，包含审计目的、审计期间、审计重点、审计方法、抽样方案（样本总量 + 抽样策略）

#### （二）访谈作业

- **功能**：基于审计方案判断是否需要访谈，逐步生成方案、问卷、访谈结果
- **输入**：审计方案、公司人员岗位职责、知识库（访谈问题库）、问题清单（非必选）
- **输出**：访谈计划（访谈清单 + 访谈策略）、访谈问卷、访谈结论
- **步骤**：
  1. 根据审计方案需访谈内容，调用人事 MCP 匹配访谈人员并输出访谈计划，守门
  2. 根据访谈计划生成访谈问卷，守门后 ATA 给各被访谈人员发送任务及问卷
  3. 线下访谈后上传访谈结果（文字记录），AI 分析访谈结论完整性并判断是否需要补充提问

#### （三）检查作业

- **功能**：基于审计方案中的审计目的、审计重点，结合资料清单库、业务流程库，生成需获取的资料清单及检查计划
- **输入**：审计方案、知识库（标准资料需求模板库）、访谈结果（非必选）、资料检查结果（非必选）
- **输出**：资料检查计划、审计底稿、问题清单
- **步骤**：
  1. 根据审计方案业务重点，结合资料清单、业务流程库输出资料检查计划，守门
  2. AI 自动生成 SQL 从数据中台获取数据，需外部获取数据的调用 API，根据检查点检查问题，生成审计底稿、问题清单（保留业务资料手动上传窗口）
  3. AI 汇总分析是否达成审计目的，若无法达成则提出资料补充获取建议及下一步审计方向

#### （四）问题确认

- **功能**：ATA 将各项问题下发至任务中心（每条问题一条任务），由岗位负责人确认问题是否属实并反馈
- **输入**：审计发现
- **输出**：经确认的审计问题
- **步骤**：
  1. 将问题清单拆分，ATA 下发各负责人确认
  2. 若反馈确认 → 进入下一步；若反馈不确认 → AI 分析反馈是否合理，合理则关闭问题并保存至知识库，不合理则输出理由并继续下发

#### （五）出具审计报告

- **输入**：项目基本资料、问题清单
- **输出**：整改建议、审计报告

## 六、功能五：离任审计

### 6.1 当前流程

风控系统立项（由 BPM/OA 审批完成后同步）→ 制定项目方案 → 发送审计通知 → 访谈阶段 → 作业阶段 → 输出初版缺陷清单 → 与被审计单位沟通缺陷情况 → 确认终版缺陷清单 → 输出个人离任审计报告和离任审计总报告 → 各业务部门跟进整改

### 6.2 业务痛点

- 审计方案无统一的知识沉淀，均依靠个人业务经验进行审计方案编制
- 人工查询整理方式效率低下
- 数据量大且多元造成数据处理困难

### 6.3 知识库与数据需求

- 历史项目审计发现清单
- 历史审计方案清单
- 公司组织架构、花名册、岗位职责
- 各个业务循环所涉及的公司各系统数据（业务数据 + 流程数据）
- 方案、访谈清单、报告模板
- 历史改善建议

### 6.4 智能体输出成果

- 标准化审计方案（含范围、目标、方法、程序）
- 审计工作底稿（半自动填充）
- 审计报告（草稿及正式版）
- 审计问题清单

### 6.5 工作流详细步骤

```
（一）审计方案生成
    ↓
（二）生成访谈问卷
    ↓
（三）生成资料清单
    ↓
（四）问题清单生成
    ↓
（五）审计问题确认
    ↓
（六）出具审计报告
```

#### （一）审计方案生成

- **功能**：根据被审计人的"职责配置"和"任职时间"，结合方案库中的历史模板与 AI 算法，自动生成详细审计方案
- **输入**：
  - (1) 被审计人基础信息（来自风控系统立项审批后推送）：事业部、离职人姓名/工号/岗位/部门、入职日期、最后工作日、组长名称、组员名称、项目名称/编号、审计期间
  - (2) 职责配置库（来自 MCP 数据调用）：人员的具体职责内容
  - (3) 审计方案库（来自知识库）：个人审计方案库（通用审计策略）+ 业务审计方案库（标准审计程序）
- **匹配逻辑**：
  - 读取被审计人岗位职责列表 → 在业务审计方案库中检索强关联业务内容 → 在个人审计方案库中检索通用/特有检查项
  - 审计期间确认：1年<本岗位时间≤5年，审计期限为离职日期向前计算三年；>5年，向前计算一年
  - 抽样方案：AI 根据业务量级和风险程度建议样本量和抽样方法
- **输出**：《离任审计实施方案》，包含被审计人信息、审计期间、审计范围（职责范围 + 业务范围）、审计方法、抽样方案

#### （二）生成访谈问卷

- **功能**：基于审计方案中的重点审计领域及被审计人岗位职责，自动生成结构化、定制化访谈提纲
- **输入**：审计方案、被审计人岗位职责、知识库（访谈问卷通用问题）
- **输出**：结构化《离任审计访谈问卷》
- **新增需求**：支持离职人是否可确认选项；可通过下拉框选择审计人员（组长/组员）确认问卷

#### （三）生成资料清单

- **功能**：基于审计方案中的审计范围，自动生成资料清单
- **输入**：审计方案、知识库（标准资料需求模板库）
- **输出**：资料需求清单
- **新增需求**：资料需求增加提供资料人员名称，同一人员多条资料合并发送至任务中心

#### （四）问题清单生成

- **功能**：对业务数据、行为数据进行验证分析，识别存在的风险场景
- **输入**：
  - 天眼查外部数据（关联关系）
  - 业务系统数据（BPM/OA/DTC 等日志与单据）
  - 人工上传线下数据（表格、截图）
  - 行为风险预警（智能体其他模块传回的结果）
  - 历史个人审计问题和业务审计问题
- **输出**：
  - 个人问题清单：商业秘密泄露、个人报销、样机使用、关联公司、资产使用等个人层面问题
  - 业务问题清单：流程漏洞、制度缺陷、经济损失等业务层面问题

#### （五）审计问题确认

- 将问题清单明细汇总，经风控人员守门确认后，输出最终版缺陷清单
- 与业务部门初步沟通确认（发布任务/邮件）
- 生成审计意见表（审计发现问题 + 审计建议 + 审计沟通对象），推送至风控系统审计作业模块

#### （六）出具审计报告

- **输入**：已确认的个人问题清单和业务问题清单、审计报告模板
- **输出**：
  - 标准化《离任审计报告书》，经风控人员守门后推送至风控系统审计报告模块
  - 传递给"整改跟踪智能体"的结构化问题列表

## 七、功能六：商业秘密保护

### 7.1 当前流程

定密和评审流程：保密员组织梳理《商业秘密信息表》及《涉密文件清单》并存证 → 保密员登录风控系统提交定密流程 → BPM/OA 审批通过 → 定密信息纳入风控系统商业秘密信息台账 → 集团评审小组组织评审

定密信息评审环节需对涉密信息类型、密级、文件清单、涉密人员范围、存证编号等的完整性、合理性、准确性进行评审。

### 7.2 业务痛点

1. 保密员提交的定密信息涉及跨事业部跨部门的多领域内容，需要充分利用业务专业知识、外部法律法规、商业秘密和知识产权侵权诉讼案件信息、商业秘密管理思维、同类部门横向比对信息，存在效率不足、专业领域和外部信息支撑不足的问题
2. 保密员提交定密流程前需反复比对已提交定密信息和现有信息，且缺少专业指导意见

### 7.3 建设目标

1. 利用智能体分析内部及外部专业信息，汇总定密信息，碳硅基协同完成定密信息评审工作
2. 在保密员提交定密流程前提供定密建议，提高保密员定密工作效率和质量

### 7.4 知识库需求

| 来源 | 内容 |
|------|------|
| 风控系统 | 保密员清单、保密项目清单 |
| 风控系统 | 《商业秘密信息表》、《涉密文件清单》 |
| 风控系统 | 部门商业秘密信息档案台账、项目商业秘密信息档案台账 |
| MDM | 员工部门架构信息（根据保密部门、保密项目匹配） |
| 制度文件 | 公司商业秘密、保密员、知识产权、业务相关管理制度 |
| 威科先行 | 外部法律法规、侵权诉讼案件信息 |
| 风控智能体 | 前期评审结果、前期定密建议 |

### 7.5 智能体输出成果

- 定密信息评审报告
- 保密员定密建议/定密信息预审报告
- 管理报告

### 7.6 主要功能

#### 功能一：保密员定密建议 / 定密信息预审

- **范围选择**：定密组织
- **前置条件**：保密员填写定密信息后点击【定密信息预审】按钮
- **风控系统需推送**：本次填写的定密信息的文件清单 + 往期定密信息及文件清单
- **赫尔墨斯输出**：定密建议预审报告（含两块内容：前期已定密信息评审结果 + 建议定密内容，与《商业秘密信息表》模板格式保持一致，便于直接导入系统）
- **保密员反馈**：选择完成接受/部分接受/不接受，并输入反馈意见
- 同一流程可多次预审，多次预审结果分块显示

#### 功能二：定密信息评审

- **范围选择**：评审组织、评审期间（暂定取消不作选择）、评审流程
- **入口**：风控系统 → 商业秘密管理 → 信息档案台账 → 部门或项目台账页面，增加【定密信息评审】按钮
- **赫尔墨斯输出**：定密评审报告
- 增加评审报告历史记录列表页面

#### 功能三：商业秘密管理情况报告

- **频率**：月度定期生成
- **内容**：公司整体商业秘密定密进度和质量情况分析和总结
  - 总体进度和数量情况：总定密组织数量、已定密组织数量、占比、总定密流程数量、总定密信息条数
  - 期间进度和数量情况：定密流程数量、定密信息条数
- **展式**：数据部分采用数据表或图表
- **总结范围可选**：全集团、某事业部、某部门/某项目
- 可查看往期管理报告（通过历史任务查看）

## 八、功能七：行为风险（员工行为分析）

对各监管系统产生的行为数据进行定期分析，识别异常行为，提前预警处理。

### 8.3 建设目标

利用智能体分析各系统行为数据，识别异常的员工行为，实现跨系统数据整合与智能分析，提升行为风险识别的效率和准确性。

### 8.4 知识库需求

| 来源 | 内容 | 用途说明 |
|------|------|----------|

| 风控系统 | 商业秘密信息、涉密文件清单 | 涉密行为关联分析 |

| 威科先行 | 外部法律法规、侵权诉讼案件信息 | 行为合规性外部基准 |
| 风控智能体 | 前期行为分析结果 | 历史分析结论回溯与趋势研判 |
| 德鲁克 | 员工岗位职责 | 行为与岗位职责匹配分析 |

### 8.5 智能体输出成果

- **行为风险分析报告**：对指定范围内员工行为数据进行分析，识别异常行为并出具报告
- **行为风险管理情况报告**：定期（月度）对公司整体行为风险管理情况进行分析和总结

### 8.6 主要功能

#### 功能一：行为风险分析

- **范围选择**：事业部、部门、岗位、人员、角色、期间
- **输出**：行为风险分析报告，包括异常行为识别、风险等级评估、关联分析结论等
- 支持查看往期分析报告（历史分析结果）

#### 功能二：行为风险管理情况总结报告

- **频率**：月度定期生成
- **内容**：
  - 监管系统覆盖范围（已接入的监管系统清单及覆盖情况）
  - 缺失范围（尚未覆盖的监管盲区）
  - 行为数据质量（各系统行为数据的完整性、及时性、准确性评估）
  - 高风险行为展示（高风险行为事件汇总与详情）
  - 高频行为问题展示（频发行为问题分类统计与趋势分析）
  - 优化方案和建议（基于分析结论的管理改进建议）
- 展示报告清单，可查看单项报告详情
- 可查看往期管理报告

#### 功能三：历史分析与管理报告查看

- 历史分析结果：查看往期行为风险分析报告，支持按时间、范围等条件检索
- 历史管理报告：查看往期管理报告，支持按时间检索

### 8.7 前台页面需求

- 定期输出行为风险管理情况报告

## 九、功能八：持续改善（审计整改跟踪）

### 9.1 产品定位

审计持续改善智能体是面向审计岗的专属智能跟踪工具，核心聚焦审计过程中发现问题的整改承接、全程跟踪、闭环管控场景，通过固定表头规范数据展示、流程化推进整改、智能提醒、全程留痕等功能，实现审计整改全流程标准化管理。

### 9.2 目标用户

| 用户类型 | 职责 |
|----------|------|
| **核心用户：审计岗人员** | 负责承接、录入、跟踪、统计审计问题整改，开展整改复核，输出跟踪报告 |
| **关联用户：整改责任人** | 推进审计问题整改落地，提交整改材料、更新整改进度 |
| **关联用户：管理层** | 查看整改跟踪进度、闭环情况及统计数据 |

### 9.3 核心价值

- **统一管理**：打造审计问题整改单一跟踪入口，整合所有审计项目发现的整改事项
- **规范标准**：通过固定表头统一审计整改数据口径、复核、闭环流程
- **效率提升**：自动化跟踪整改进度、智能提醒逾期事项，减少人工手动跟踪统计工作量
- **闭环管控**：全程记录审计问题从录入、下发、整改、复核到闭环的全流程

### 9.4 核心功能模块

#### 9.4.1 问题录入功能

- 从风控系统（内控评价/廉洁监察/商业秘密/行为风险）整改跟踪模块自动抓取问题清单
- 风控系统整改跟踪支持手动单条录入、Excel 批量导入（按固定表头模板下载填写后导入），导入后可手动编辑修改
- 支持问题作废操作（需填写作废原因，留痕备查）

**校验规则**：必填字段不能为空；问题编号唯一；整改期限不能早于录入日期

**状态说明**：

| 状态 | 含义 |
|------|------|
| 问题待推送 | 审计问题清单同步至整改智能体后（组长确认下发前） |
| 计划待提交 | 审计组长下发整改任务后，整改部门还未上传整改计划/方案 |
| 计划待审批 | 整改部门上传整改计划/方案，审计跟进人员还未审核 |

| 整改完成 | 审计跟进人员已完成整改证据的审核，并确认通过 |

#### 9.4.2 整改任务下发功能

1. 项目组长在赫尔墨斯确认问题录入无误后，可一键"下发整改任务"
2. 系统自动将整改任务推送至对应部门负责人（任务中心）
3. 同时自动推送给审计跟进人"计划/方案审核任务"
4. 下发后整改状态自动更新为"计划待提交"，部门负责人收到 Elink 通知
5. 部门负责人在任务中心提交具体责任整改人、审计整改计划/方案 → 状态更新为"计划待审核"
6. **AI 初审**：AI 对业务部门负责人上传的附件进行初审并给出意见
7. 审计整改跟进人进行二次确认
8. 确认通过后，赫尔墨斯再向业务部门推送"审计整改任务"（AI 分析整改计划自动生成整改目标和任务描述）→ 状态更新为"整改答复待提交"

#### 9.4.3 整改跟踪功能

- 问题整改责任人在任务中心提交审计整改证据，须严格按照提交的审计整改计划/方案执行
- 审计岗可按多种条件筛选查看所有审计问题整改进度
- 支持单个问题详情查看（含全流程操作记录、整改材料、复核意见等）
- 审计岗可一键催办，发送督促消息，记录督促情况

#### 9.4.4 整改复核与闭环功能

**复核流程**：

1. 责任人提交整改完成 → 状态更新为"已整改待复核"
2. **AI 初审核**：AI 依据前期上传的整改计划/方案，对责任人上传的附件进行初审判断
3. AI 初审核通过后自动向审计岗 Elink 发送复核提醒并向任务平台推送复核任务
4. 审计岗查看整改说明、审核整改佐证材料，选择"复核通过"或"复核不通过"

**闭环管理**：
- 复核通过 → 状态自动更新为"整改完成"，完成闭环
- AI 复核或审计复核不通过 → 需填写退回原因 → 状态更新为"退回重改" → 向责任人发送退回提醒
- 若退回次数达 3 次及以上 → 同步通知所在部门负责人

**批量复核**：可增加筛选功能，对不同类型问题一键复核完成

**复核留痕**：所有复核操作永久留痕，不可篡改，满足审计合规核查需求

#### 9.4.5 问题归档功能

- 整改闭环后系统自动归档，归档后不可编辑，仅可查看
- 支持按多种条件筛选查看归档事项
- 归档数据可按筛选条件导出，用于存档、复盘分析

### 9.5 通用支撑模块

#### 9.5.1 统计分析功能

- **自定义统计**：按时间段、审计项目、责任部门、问题类型、风险等级、整改状态等条件统计
- **统计指标**：审计问题总数、整改中数量、已完成数量、逾期数量、退回重改数量、整改完成率、逾期率
- **报表生成**：自动生成审计整改跟踪统计报表，支持 Excel、PDF 格式导出
- **可视化展示**：通过柱状图、饼图、折线图直观展示整改进度、完成情况、各部门整改效率

#### 9.5.2 操作日志功能

- 所有操作（问题导入、下发、进度更新、复核、退回、权限变更、数据导出等）均记录操作日志
- 日志包含操作人、操作时间、操作内容、操作结果
- 支持按操作人、操作时间、操作类型筛选查询
- 日志不可删除、不可篡改，满足审计合规核查需求
- 可通过操作日志追溯问题原因，明确责任主体

### 9.6 权限控制

- 项目组长：可看到负责项目下所有审计问题清单
- 项目成员：仅能看到被分配跟进的问题清单
- 事业部风控负责人：可看到该事业部所有问题清单
- 风控负责人：可看到所有问题清单

## 十、公共能力需求

### 10.1 统一权限管理（RBAC）

| 角色 | 权限范围 |
|------|----------|
| 集团 | 可查看所有数据 |
| 科沃斯 | 仅可查看涉及公司为科沃斯的数据 |
| 添可 | 仅可查看涉及公司为添可的数据 |

- 在所有涉及筛选的界面中，依据登录用户的角色不同，仅能看到对应角色的数据
- 手动添加案件信息时各角色只能添加所属公司对应的数据
- 任务中心界面也遵从上述权限管理

### 10.2 知识库体系

系统按业务阶段划分多个独立知识库，各功能模块在各工作流阶段调用对应的知识库内容：

| 阶段 | 知识库内容 | 核心用途 |
|------|-----------|---------|
| **初筛阶段** | 组织架构、人员名单、岗位职责、客户/供应商清单、内部制度、外部法规 | 判断案件性质与严重程度 |
| **调查方案阶段** | 类似案件法条、业务系统信息、过往舞弊案件及处理方案 | 生成调查方向与方案 |
| **分析报告阶段** | 过往调查报告、报告模板及格式要求 | 生成规范报告 |
| **处置分流阶段** | 制度文件、追责审批流程、组织架构及分权 | 追责意见生成 |
| **处罚执行阶段** | 黑名单制度、赔偿协议模板、处罚公告模板、人员架构 | 公告与协议撰写 |

各功能模块的知识库独立管理，支持离线本地维护与增量更新。详见各模块章节的"知识库内容"说明。

### 10.3 A2A 智能体间通信

系统通过 A2A（Agent-to-Agent）协议与以下外部智能体协作：

| 智能体 | 交互场景 |
|--------|----------|
| **龟宝** | 承接员工处罚落实与跟踪；HR相关案件转交 |
| **西塞罗** | 民事纠纷案件推送；处罚协议审核；定密评审辅助 |
| **波特** | 供应商扣款跟踪 |

A2A 交互采用统一的任务消息格式，包含任务ID、来源/目标智能体、操作指令、案件引用及业务载荷，支持回调确认与状态同步。

### 10.4 风控系统对接

- 风控系统通过特定节点调用赫尔墨斯智能体进行辅助
- 守门完成后自动将风控系统内对应字段进行补充
- 字段映射关系见廉洁监察模块 2.6.5 节
- 交互按钮状态跟随赫尔墨斯内事件状态变化
- 支持 WebSocket 推送阶段状态变更

### 10.5 操作日志与审计

- **全程留痕**：所有操作均记录操作日志，不可删除、不可篡改
- **日志字段**：操作人、操作时间、操作内容、操作结果、IP 地址
- **日志查询**：支持按操作人、操作时间、操作类型筛选
- **安全等级**：等保二级要求，审计日志保留 >= 6 个月

### 10.6 输出物通用规范

- 所有输出物均支持**方案变更**和**划词调整**两种编辑功能
- 导出格式限定在 Word (docx)、Excel (xlsx) 等常用格式
- 每个工作流阶段均设置**碳基守门**（human-in-the-loop）环节

## 十一、智能体（Agent）设计规范

本章定义赫尔墨斯系统中各功能模块智能体的通用设计规范，涵盖智能体的角色定义、决策原则、安全约束、评估标准和跨智能体协作机制。

### 11.1 Agent 角色定义

系统内每个功能模块对应一个或多个 Agent 节点，每个 Agent 具有明确的角色身份与行为边界：

| Agent 节点 | 所属模块 | 角色身份 | 核心任务 | 输出特征 |
|-----------|----------|----------|----------|----------|
| **初筛 Agent** (intake) | 廉洁监察 | 案件初审官 | 评估线索可信度、判断是否立案、分流决策 | 结构化判断 + 置信度 + 理由 |
| **调查方案 Agent** (investigation) | 廉洁监察 | 调查策略师 | 匹配类似案例、生成调查方向与方案 | 可执行方案 + 参考案例 |
| **分析报告 Agent** (analysis) | 廉洁监察 | 数据分析师 | 多维数据碰撞、证据链构建、报告撰写 | 数据分析报告 + 案件结论 |
| **处置分流 Agent** (disposition) | 廉洁监察 | 法律顾问 | 追责定性、法律路径推荐、处罚建议 | 法律分析 + 追责意见 |
| **处罚执行 Agent** (enforcement) | 廉洁监察 | 执行协调员 | 公告撰写、协议生成、跨系统任务派发 | 格式化文档 + A2A 任务 |
| **风险规则 Agent** | 风险监控 | 风控规则师 | 场景识别、规则生成、SQL 编写 | 可执行规则 + SQL |
| **风险分析 Agent** | 风险监控 | 风险分析师 | 异常筛选、主体合并、风险定性 | 分析报告 + 风险等级 |
| **审计方案 Agent** | 内控/专项/离任审计 | 审计规划师 | 方案生成、抽样策略、人员匹配 | 审计方案文档 |
| **审计检查 Agent** | 内控/专项/离任审计 | 审计执行师 | 缺陷识别、底稿填充、打分评价 | 缺陷清单 + 底稿 |
| **离任审计 Agent** | 离任审计 | 离职审查专家 | 个人/业务问题双轨分析、审计期间计算、MCP 职责匹配 | 双轨问题清单 + 审计意见表 |
| **访谈 Agent** | 多模块共享 | 访谈协调师 | 访谈人员匹配、问卷生成、结论完整性分析 | 访谈计划 + 问卷 + 结论 |
| **定密评审 Agent** | 商业秘密 | 保密评审专家 | 定密建议、完整性审查、横向比对 | 评审报告 |
| **行为风险分析 Agent** | 行为风险 | 行为分析专家 | 跨系统行为数据整合、异常行为识别、风险等级评估 | 行为风险分析报告 + 管理报告 |
| **整改跟踪 Agent** | 持续改善 | 整改督导员 | 计划初审、证据复核、逾期提醒 | 复核意见 + 统计报表 |

### 11.2 Agent 决策原则

所有 Agent 在自主判断时必须遵循以下通用原则：

| 原则 | 说明 | 违反时的处理 |
|------|------|-------------|
| **证据驱动** | 任何判断必须引用知识库中的具体条文、案例或数据作为依据 | 碳基守门时应驳回，要求补充依据 |
| **可解释性** | 每个输出结论必须附带"为什么这样判断"的理由说明 | 无理由的输出视为不完整 |
| **保守倾向** | 在依据不充分时，宁可建议"暂缓处理/补充信息"而非激进决策 | Agent 主动标记为"低置信度"并建议人工主导 |
| **人主导裁决** | Agent 提供建议，最终决定权始终在碳基（人类）手中 | Agent 输出应包含"建议"措辞而非"指令"措辞 |
| **合规优先** | 法律合规要求绝对优先于效率或便利性考量 | 任何可能触碰合规红线的建议必须显式标注风险 |
| **过往案例参考** | 相似历史案例的处理结果应作为重要参考，但不机械套用 | 需标注案例差异点，由碳基判断适用性 |

### 11.3 置信度与不确定性处理

Agent 在每个判断输出中必须包含**置信度标记**，用于指导碳基守门的关注重点：

| 置信度等级 | 含义 | Agent 行为 | 碳基守门动作 | 示例场景 |
|-----------|------|-----------|-------------|----------|
| **高 (≥85%)** | 依据充分、案例相似度高、无歧义 | 正常输出建议，标记"建议采纳" | 快速确认即可 | 明确违反明文制度的舞弊行为 |
| **中 (60%-85%)** | 有一定依据但存在不确定因素 | 输出建议 + 标注不确定点 + 提供替代方案 | 需仔细审核不确定点 | 涉及灰色地带、制度未明确覆盖 |
| **低 (40%-60%)** | 依据薄弱、案例差异大、关键信息缺失 | 降级为"辅助分析"，输出初步方向而非明确建议 | 人工主导决策，Agent 仅提供参考 | 新型舞弊手法、跨领域复杂案件 |
| **无法判断 (<40%)** | 缺乏足够信息或超出知识范围 | 明确声明"超出能力范围"，列出需要补充的信息 | 完全人工决策，Agent 仅辅助信息整理 | 涉及外国法律、专业领域问题 |

**不确定性触发规则**：
- 知识库检索返回相似度 < 0.7 → 标记"参考案例有限"
- 关键输入字段缺失超过 30% → 标记"信息不完整"
- 存在多个矛盾解释且无法排除 → 标记"存在歧义"
- 涉及金额超过阈值（如 100 万元）→ 强制标记为"高风险，建议升级审核"

### 11.4 Agent 安全约束（硬性底线）

以下规则 Agent **绝对不能违反**，应通过 System Prompt 和输出校验双重保障：

| 约束类型 | 具体规则 |
|----------|----------|
| **法律底线** | 不得建议或暗示任何违反中国法律法规的行为；不得建议销毁证据、作伪证、私下解决刑事案件 |
| **权限边界** | 不得越权操作外部系统（如直接修改风控系统数据、直接发起 OA 审批）；所有外部写操作必须经过碳基守门 |
| **隐私保护** | 不得在输出中暴露其他无关人员的个人信息；案例引用时必须去标识化（姓名用"张某"替代） |
| **利益冲突** | 不得在处理涉及自身开发团队所属事业部的案件时给出倾向性建议 |
| **歧视禁止** | 不得基于性别、年龄、民族、地域等因素在同类案件中给出差异化建议 |
| **虚假信息** | 不得编造不存在的法条、案例或数据来支撑结论（Anti-Hallucination） |

**输出校验机制**：
- LLM 输出后自动扫描敏感词（如真实姓名、身份证号、银行卡号）
- 引用法条自动校验（与知识库中法条库比对）
- 金额/数字字段进行合理性检查（如追责金额不应 > 涉案金额）

### 11.5 Agent 异常处理策略

| 异常场景 | 检测方式 | Agent 行为 | 系统行为 |
|----------|----------|-----------|----------|
| **LLM 返回空/乱码** | 输出长度 < 50 字符 或 格式校验失败 | — | 自动重试 1 次 → 仍失败则进入 human_intervention 节点 |
| **LLM 超时 (30s)** | 调用超时 | — | 自动切换备用 LLM (通义千问) 重试 → 仍超时则进入 human_intervention |
| **幻觉检测** | 输出中的法条/案例编号在知识库中不存在 | — | 标记为"待验证"，碳基守门时强制人工审核 |
| **结构化输出不完整** | JSON Schema 校验失败（必填字段缺失） | — | 自动重试（附校验错误信息）→ 3 次失败后降级为自由文本输出 |
| **知识库检索失败** | PGVector 连接超时或返回空 | 使用 LLM 内部知识完成任务，但显式标注"未检索到相关文档" | 触发 P2 告警 |
| **外部 A2A Agent 无响应** | 超过回调超时时间 (5min) | — | 任务保留队列，每 5min 重试 → 30min 仍无响应进入 human_intervention |

### 11.6 Agent 评估标准

各模块 Agent 需满足以下质量基线方可上线：

| 评估维度 | 指标 | 基线 | 测量方式 |
|----------|------|------|----------|
| **准确率** | 碳基守门"直接通过"比例 | ≥ 70% | 守门审批记录统计 |
| **驳回率** | 碳基守门"驳回修改"比例 | ≤ 20% | 守门审批记录统计 |
| **严重错误率** | 碳基守门"完全否定"比例 | ≤ 5% | 守门审批记录统计 |
| **置信度诚实性** | Agent 标记"低置信度"时的实际错误率 | ≤ 30%（即低置信度输出中仍有 70% 可接受） | 人工抽检 |
| **效率提升** | 与纯人工相比的处理时间缩短 | ≥ 50% | 案件全流程耗时对比 |
| **用户满意度** | 碳基用户对 Agent 输出的主观评分 | ≥ 4.0/5.0 | 阶段完成后评分弹窗 |

**持续评估机制**：
- 每周自动生成 Agent 质量报告（按模块/阶段/Agent 节点维度）
- 连续 2 周驳回率 > 30% 的 Agent 节点 → 触发 Prompt 优化
- 连续 4 周驳回率 > 30% → 触发该阶段重新设计

### 11.7 跨 Agent 上下文传递规范

当一个模块/阶段的 Agent 完成任务并交接给下游时，需要传递标准化的上下文信息：

```json
{
  "source_agent": "intake",
  "target_agent": "investigation",
  "case_ref": "GZ2026051901",
  "confidence": "high",
  "key_findings": [
    "举报人提供了明确的供应商名称和转账记录",
    "涉案金额约 50 万元",
    "涉及采购部员工张某与供应商 XX 公司的利益输送"
  ],
  "outstanding_questions": [
    "张某在 XX 公司的持股比例待确认",
    "2024 年 Q3 的采购定价是否偏离市场价"
  ],
  "suggested_focus": [
    "重点审查张某经手的 2023-2024 年全部采购订单",
    "调取 XX 公司的工商信息和股东结构"
  ],
  "evidence_summary": {
    "documents": 12,
    "audio_files": 2,
    "images": 5,
    "key_evidence_ids": ["DOC-001", "AUD-002"]
  },
  "risk_flags": ["涉及金额较大", "供应商为关联公司"],
  "transferred_at": "2026-05-19T10:30:00Z"
}
```

**传递原则**：
- 上游 Agent 的关键发现和未解决问题是下游 Agent 的核心输入
- 置信度标签随上下文传递，下游 Agent 据此调整分析深度
- 风险标记在整个链路中累积，最终处置阶段汇总所有标记

### 11.8 Agent Prompt 策略

| 策略 | 说明 |
|------|------|
| **角色锚定** | 每个 Agent 的 System Prompt 首句明确定义其专业身份（如"你是一位有 15 年经验的反舞弊调查专家"） |
| **知识注入** | 每个阶段启动时，从对应知识库检索 Top-K 相关文档注入 Prompt |
| **历史摘要注入** | 将上游阶段的上下文传递 JSON 作为当前 Prompt 的一部分 |
| **输出格式约束** | 每个 Agent 定义 output_schema (Pydantic)，通过 Function Calling / JSON Mode 保证结构化输出 |
| **负面约束** | System Prompt 末尾附 11.4 节的安全约束要点 |
| **Few-shot 示例** | 关键决策节点附带 2-3 个正面 + 1 个反面示例 |
| **版本管理** | Prompt 按 YAML 文件版本化存储（v1/v2 目录），prompt_config.yaml 控制各节点当前生效版本 |
