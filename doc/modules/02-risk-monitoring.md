# 风险监控模块需求说明书

> 文档版本：V3.0
> 所属系统：科沃斯-赫尔墨斯AI系统
> 模块名称：风险监控（主动风险扫描）

---

## 1. 项目背景与业务痛点

### 1.1 现状问题

当前风险监控体系主要依赖人工识别与处置，存在以下核心痛点：

| 痛点 | 描述 |
|------|------|
| **人工识别效率低，响应滞后** | 无标准化指标的风险场景依赖人工判断，受经验、精力、工作时间限制，易漏判、误判，无法 7×24 小时覆盖，风险发现与处置不及时 |
| **指标覆盖不足，监控范围有限** | 现有指标依赖人工提炼与历史经验，数量少、覆盖窄，存在大量管控盲区，无法满足全业务、全流程、常态化风险监控要求 |

### 1.2 解决思路

引入 AI 智能体，构建自动化、全覆盖、可闭环的风险监控体系，实现指标智能生成、7×24 小时监控、异常自动分析、风险自动定性、报告自动输出、结果自动推送与迭代优化。

---

## 2. 建设目标

### 2.1 总体目标

以 AI 智能体为核心，围绕指标拓展、流程自动化、风险精准化、处置闭环化，实现风险监控全流程智能化升级，为风控决策提供及时、准确、可追溯的数据支撑。

### 2.2 具体目标

| 序号 | 具体目标 |
|:---:|----------|
| 1 | 实现 7×24 小时无人值守自动监控，实时捕捉异常 |
| 2 | 自动完成异常数据分析、原因定位、影响评估 |
| 3 | 自动完成风险等级、风险类型判定 |
| 4 | 自动输出日/周/月标准化风险报告 |
| 5 | 自动推送至相关处置模块并形成处置闭环 |

## 6. 具体工作流程

工作流由 6 个阶段构成，按以下顺序执行：

```
[6.1] 风险规则清单生成 (AI + 人工审核)
    ↓
[6.2] 风险异常数据生成 (AI 跑数 + AI 初核 + 人工复核)
    ↓
[6.3] 风险分析与主体合并 (AI 自动聚合)
    ↓
[6.4] 风险自动定性 (AI 判定 + 人工可修正)
    ↓
[6.5] 风险结果自动推送
    ↓
[6.6] 处置结果回流与指标迭代 (闭环核心)
```

---

### 6.1 风险规则清单生成（AI + 人工审核）

#### 功能概述

AI 自动生成风险场景、计算规则、SQL 语句，经人工审核后入库，形成标准化的风险清单知识库。

#### 输入

| 输入项 | 说明 | 来源 |
|--------|------|------|
| 风险规则 | 事业部、渠道、业态、部门、岗位、一/二/三级场景 | 业务部门 / AI生成 |
| 数据库字段及含义 | 业务数据库的表结构、字段定义 | 数据中台 |
| 历史案件判例 | 过往审计、风险案例 | 历史知识库 |
| 人工自定义场景 | 用户手动输入或上传的风险场景（可选） | 业务人员 |

#### 处理逻辑

| 步骤 | 动作 | 说明 |
|:---:|------|------|
| 1 | **选择模式** | 用户选择清单生成模式：清单上传（批量导入现有规则）/ 人工录入（逐条手动录入） |
| 2 | **AI 生成三级场景** | AI 根据业务数据字段和历史案例，从一级场景逐步细化至三级场景 |
| 3 | **AI 生成计算规则** | 基于三级场景，AI 自动推导异常判定规则和阈值 |
| 4 | **AI 生成 SQL 语句** | AI 将计算规则转化为可执行的 SQL 查询语句 |
| 5 | **SQL 测试验证** | AI 在测试环境自动执行 SQL，验证语法正确性及可执行性 |
| 6 | **人工审核** | 审核通过 → 入库；审核不通过 → 退回修正或作废 |
| 7 | **形成知识库** | 通过审核的规则存入风险清单知识库，供后续跑数使用 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 风险清单知识库 | 包含标准字段的结构化规则清单，详见 4.1 节字段定义 |

#### 流程图

```
开始
  │
  ▼
选择模式：[清单上传] ──┬── [人工录入]
                      │
                      ▼
              AI 生成三级场景
                      │
                      ▼
              AI 生成计算规则
                      │
                      ▼
              AI 生成 SQL 语句
                      │
                      ▼
              AI 在测试环境验证 SQL
                      │
                      ▼
            ┌── 人工审核 ──┐
            │              │
          通过            不通过
            │              │
            ▼              ▼
         入库         修正 / 作废
            │
            ▼
     形成风险清单知识库
            │
            ▼
          结束
```

### 6.2 风险异常数据生成（AI 跑数 + AI 初核 + 人工复核）

#### 功能概述

按照风险清单知识库中的规则，AI 按配置频率自动执行 SQL 跑出异常数据，经过 AI 初核和人工复核双重校验，确保异常数据准确有效。

#### 输入

| 输入项 | 说明 | 来源 |
|--------|------|------|
| 风险清单知识库 | 标准化的规则清单，含SQL、阈值、频率 | 6.1 阶段产出 |
| 业务数据 | 各业务系统实时数据 | 业务数据库 |
| 外部数据 | 企查查、舆情等外部信息 | 外部数据接口 |

#### 处理逻辑

| 步骤 | 动作 | 说明 |
|:---:|------|------|
| 1 | **AI 执行跑数** | AI 根据风险清单中的 SQL 语句和监控频率（日/周/月），自动连接数据库执行查询，生成异常数据明细 |
| 2 | **AI 初核** | AI 对跑出的异常数据进行初步审核：剔除明显属于正常业务的数据（如符合业务规律的节假日波动），并对保留的异常项标注异常说明和可能原因 |
| 3 | **人工二次复核** | 业务人员或风控人员对 AI 初核后的异常数据进行复核确认 |
|  | → **复核通过** | 标记为有效异常，进入下一阶段分析 |
|  | → **复核不通过** | 根据实际情况调整规则（优化 SQL/阈值）/ 指标作废 / 回流至 6.1 阶段优化 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 风险预警异常数据明细 | 每条异常数据的详细记录 |
| 风险预警异常数据透视表 | 按多维度汇总的异常数据视图 |
| 风险预警异常数据分析报告 | AI 初核后生成的异常分析说明报告 |

#### 流程图

```
开始
  │
  ▼
AI 按 SQL 与监控频率自动执行跑数
  │
  ▼
生成异常数据明细
  │
  ▼
AI 初核：剔除正常业务 + 标注异常说明
  │
  ▼
┌── 人工二次复核 ──┐
│                   │
通过               不通过
│                   │
▼                   ├── 调整规则 (优化 SQL/阈值)
进入下一阶段        ├── 指标作废
                    └── 回流至 6.1 优化
```

### 6.3 风险分析与主体合并（AI 自动聚合）

#### 功能概述

AI 自动识别分析主体（如同一联系人、同一企业），将同一主体的多条异常预警合并，统一分析口径，避免重复预警和分散判断。

#### 输入

| 输入项 | 说明 | 来源 |
|--------|------|------|
| 异常风险明细 | 复核通过后的异常数据明细 | 6.2 阶段产出 |
| 异常风险明细透视表 | 多维度汇总的异常数据 | 6.2 阶段产出 |

#### 处理逻辑

| 步骤 | 动作 | 说明 |
|:---:|------|------|
| 1 | **AI 识别分析主体** | AI 根据联系方式、联系人、地址、统一社会信用代码等信息，自动识别并聚类分析主体 |
| 2 | **合并重复预警** | 同一主体产生的多条异常预警自动归并，避免对同一问题多次告警 |
| 3 | **标记涉及指标** | 对合并后的主体，标记所有涉及的异常指标及其数量 |
| 4 | **识别风险要素** | AI 进一步识别该主体的风险行为、风险业务、影响范围、涉及金额等关键信息 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 按分析主体合并后的风险透视表 | 以主体为维度的风险汇总视图 |
| 单主体风险分析报告 | 每个分析主体的详细风险分析报告，含风险行为、业务、影响范围、涉及金额 |

#### 流程图

```
开始
  │
  ▼
AI 获取异常风险明细数据
  │
  ▼
AI 识别分析主体（联系方式 / 联系人 / 地址等）
  │
  ▼
同一主体多条异常 → 自动合并
  │
  ▼
标记该主体涉及的所有异常指标
  │
  ▼
识别风险行为、风险业务、影响范围、涉及金额
  │
  ▼
输出：按主体合并的风险透视表 + 单主体分析报告
  │
  ▼
结束
```

### 6.4 风险自动定性（AI 判定 + 人工可修正）

#### 功能概述

AI 基于单主体风险分析报告和历史审计案例知识库，自动判定风险类型、风险等级、严重性、广泛性、影响程度，并给出处置建议。人工可对 AI 判定结果进行修正。

#### 输入

| 输入项 | 说明 | 来源 |
|--------|------|------|
| 单主体风险分析报告 | 每个分析主体的风险分析详情 | 6.3 阶段产出 |
| 历史审计案例知识库 | 过往审计案例、定性标准、处置经验 | 知识库 |

#### 处理逻辑

AI 自动输出以下判定维度：

| 判定维度 | 说明 | 评估依据 |
|----------|------|----------|
| **风险类型** | 合规风险、舞弊风险、商业秘密风险、运营风险、财务风险等 | 业务特征 + 历史案例匹配 |
| **风险等级** | 高 / 中 / 低 | 综合评分 |
| **严重性** | 风险可能造成的损失程度 | 岗位级别、涉及金额 |
| **广泛性** | 风险波及的业务范围 | 涉及部门数、业务线数 |
| **影响程度** | 综合岗位、金额、业务范围、发生频次评估 | 岗位 + 金额 + 范围 + 频次 |
| **处置建议** | AI 基于历史案例给出的处理方向建议 | 相似案例处置方案推荐 |

#### 人工修正机制

| 场景 | 处理方式 |
|------|----------|
| AI 判定合理 | 直接确认通过 |
| AI 判定偏差 | 人工手动修正判定结果，修正记录留存备查 |
| AI 无法判定 | 标注待人工判定，并记录无法判定的原因供模型优化 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 风险定性报告及处理建议 | 含风险类型、等级、严重性、广泛性、影响程度、处置建议的完整报告 |

### 6.5 风险结果自动推送

#### 功能概述

将已完成定性的风险结果，根据风险类型和涉及部门，自动分发至对应的处置模块或业务部门智能体。

#### 推送对象

| 推送目标 | 说明 | 推送内容 |
|----------|------|----------|
| **业务部门智能体** | 推送至具体业务部门的 AI 智能体，由业务部门自主处置 | 相关风险预警及定性报告 |
| **廉洁监察模块** | 涉及廉洁、腐败、舞弊类风险推送至廉洁监察系统 | 舞弊/廉洁类风险定性报告 |
| **内控内审模块** | 涉及内部控制、流程缺陷类风险推送至内控内审系统 | 内控/流程类风险定性报告 |
| **商业秘密模块** | 涉及商业秘密泄露类风险推送至商业秘密保护系统 | 商业秘密类风险定性报告 |

#### 推送规则

| 规则项 | 说明 |

| 接收确认 | 推送目标需返回接收确认，未确认则自动重试 |
| 推送记录 | 所有推送操作记录归档，含推送时间、目标、内容、状态 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 推送记录 | 包含推送时间、推送目标、推送内容、接收状态、处理进度的完整日志 |

### 6.6 处置结果回流与指标迭代（闭环核心）

#### 功能概述

各处置模块处理完成后，将处置结果回流至风险监控模块。AI 根据处置反馈优化规则和阈值，减少误报，持续提升风险识别准确率。这是实现闭环管理的核心环节。

#### 输入

| 输入项 | 说明 | 来源 |
|--------|------|------|
| 各模块处置结果 | 各处置模块完成处理后的反馈 | 6.5 推送目标模块 |
| 整改情况 | 风险整改的详细过程和结果 | 业务部门/处置模块 |

#### 处理逻辑

| 步骤 | 动作 | 说明 |
|:---:|------|------|
| 1 | **记录处置状态** | 对每项风险记录其处置状态：已处置 / 整改中 / 无需处置 / 误报 |
| 2 | **误报 / 失效指标处理** | 对标记为"误报"或"指标失效"的规则，经人工确认后，启动优化流程 |
| 3 | **规则优化** | 根据误报反馈，AI 自动或人工手动调整 SQL 语句、阈值参数等 |
| 4 | **知识库更新** | 更新风险清单知识库中的规则，持续提升准确率 |
| 5 | **迭代记录** | 记录每次指标优化的前后对比和优化原因 |

#### 处置状态定义

| 状态 | 说明 | 后续动作 |
|------|------|----------|
| 已处置 | 风险已完成处置，问题已解决 | 归档，可作为正样本用于模型训练 |
| 整改中 | 风险正在整改，尚未完成 | 持续跟踪，按周期更新状态 |
| 无需处置 | 经评估无需采取处置措施 | 归档，标记原因 |
| 误报 | AI 规则误判，实际非风险 | 启动规则优化流程，更新知识库 |

#### 输出

| 输出项 | 说明 |
|--------|------|
| 风险处置闭环台账 | 记录每项风险的完整处置链路：发现 → 分析 → 定性 → 推送 → 处置 → 归档 |
| 指标优化迭代记录 | 记录每次规则/阈值优化的变更历史、优化原因、效果对比 |

#### 流程图

```
开始
  │
  ▼
接收各模块处置结果、整改情况
  │
  ▼
┌── 记录处置状态 ──┐
│                    │
│  ┌ 已处置          │
│  ├ 整改中          │
│  ├ 无需处置        │
│  └ 误报 ──────────┼──┐
│                    │  │
└────────────────────┘  │
                        │
                      [误报 / 失效指标]
                        │
                        ▼
                  ┌── 人工确认 ──┐
                  │               │
                确认误报         非误报
                  │               │
                  ▼               ▼
           优化 SQL / 阈值    更新状态
                  │
                  ▼
            更新风险清单知识库
                  │
                  ▼
            记录优化迭代历史
                  │
                  ▼
              结束
```

## 7. 端到端流程总览

### 7.1 全流程数据流转

| 阶段 | 输入 | 处理 | 输出 |
|:---:|------|------|------|
| 6.1 | 风险规则、数据库字段、历史案例、自定义场景 | AI 生成场景→规则→SQL，人工审核 | 风险清单知识库 |
| 6.2 | 风险清单知识库、业务数据、外部数据 | AI 执行 SQL 跑数，AI 初核，人工复核 | 异常数据明细及透视表、分析报告 |
| 6.3 | 异常风险明细、透视表 | AI 识别主体，合并重复预警 | 按主体合并透视表、单主体分析报告 |
| 6.4 | 单主体分析报告、历史案例知识库 | AI 判定风险类型/等级/程度，人工可修正 | 风险定性报告及处理建议 |
| 6.5 | 风险定性报告 | 按类型分发至对应处置模块 | 推送记录 |
| 6.6 | 各模块处置结果、整改情况 | 记录状态，优化规则，更新知识库 | 处置闭环台账、优化迭代记录 |

### 7.2 闭环价值

```
风险规则清单生成 ──→ 异常数据生成 ──→ 分析主体合并
      ↑                                       │
      │                                       ▼
      │                                 风险自动定性
      │                                       │
      │                                       ▼
      │                                 风险结果推送
      │                                       │
      │                                       ▼
      └────────── 处置结果回流 ──────── 处置模块处理
                          闭环
```

通过 6.6 阶段的回流机制，每次误报和失效指标的发现都会反哺到 6.1 阶段的规则优化中，形成正向循环，不断提升系统准确率和覆盖面。
