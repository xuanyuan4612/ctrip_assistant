## 一、当前流程

### 1.1 主流程

内控评价当前主流程如下：

```
风控系统立项 → 制定项目方案 → 发送审计通知 → 访谈阶段 → 测试阶段（穿行测试 & 有效性测试）→ 输出各循环内控缺陷（设计缺陷 + 执行缺陷）→ 按照评分标准打分 → 输出内控评价报告 → 各业务部门跟进整改
```

### 1.2 详细步骤

1. 风控系统立项
2. 制定项目方案
3. 发送审计通知，根据公司内控评价标准矩阵、公司现有制度分循环开展内控评价工作
4. 访谈阶段：确定访谈人员、编制访谈问卷、访谈实施、资料收集补充
5. 测试阶段：穿行测试 & 有效性测试，收集资料、实施测试
6. 共 19 个业务循环，输出各循环内控缺陷，分设计缺陷和执行缺陷：目前各有一张底表作为底稿，根据内控评价标准矩阵填写各循环的现行控制活动和测试结果
7. 按照评分标准进行打分
8. 输出内控评价报告
9. 各业务部门跟进整改

## 二、业务痛点

### 2.1 制度设计的有效性缺乏高效监控及评价

公司制度体系的设计质量缺少持续、自动化的监控手段，制度设计是否存在漏洞、是否与控制目标匹配、是否覆盖关键风险场景，主要依赖人工判断和经验积累，缺乏系统化的评价机制。

### 2.2 流程执行的有效性缺乏高效监控及评价

控制活动在实际业务中的执行情况难以实时追踪，执行偏差、执行缺失等问题发现滞后，多数情况下需要在审计周期结束后才能通过抽样测试发现问题。

### 2.3 全业务流程内控评价耗时长、成果易被忽视

覆盖 19 个业务循环的全流程内控评价项目需要投入大量人力和时间，但重大项目发现比例不高，产出成果容易受到管理层忽视，投入产出比有待提升。

## 三、业务改善点

### 3.1 满足外部监管与审查需求

通过系统化、标准化的内控评价流程，确保企业内部控制体系持续符合《企业内部控制基本规范》及外部监管要求，为外部审计、监管检查提供有力支撑。

### 3.2 多维度缺陷评价

及时从不同维度（主体维度、循环维度、岗位维度）对设计缺陷和执行缺陷进行评价和归因分析，帮助管理层精准定位内控薄弱环节。

### 3.3 大幅缩减业务时间

将内控评价作为风控常规作业融入日常运营，通过智能化手段将评价周期从数月压缩至数周甚至更短，提升审计资源的利用效率。

### 3.4 提高缺陷整改沟通效率

通过结构化缺陷清单、自动分派整改任务、在线确认与跟踪，打通审计部门与业务部门之间的信息壁垒，加快缺陷确认和整改闭环速度。

- **岗位维度**：按岗位 / 部门归集缺陷分布情况

## 七、具体工作流程

### 7.1 立项

#### 7.1.1 流程描述

在辛顿系统立项，填写必要字段后，项目信息同步至智能体，并传递至风控系统。

#### 7.1.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 人工输入立项信息 |
| 输出 | 项目信息同步至风控系统 |

#### 7.1.3 必填字段

| 字段名称 | 说明 |
|----------|------|
| 项目名称 | 内控评价项目的名称 |
| 审计目的 | 本次内控评价的目标和范围说明 |
| 工作年度 | 评价所属年度 |
| 被审计单位 | 被评价的公司主体（可多选） |
| 审计区间 | 评价覆盖的时间范围 |
| 项目组组长 | 审计项目负责人 |
| 项目组组员 | 审计项目参与人员 |
| 标准控制活动选择 | 选择适用的标准控制活动底表 |

#### 7.1.4 规则说明

- **项目编号**：参考风控系统的项目编号生成规则自动生成，无需人工输入
- **标准控制活动选择**：在风控系统中选择对应版本的标准控制活动底表，同步至后续流程

### 7.2 审计方案生成

#### 7.2.1 流程描述

审计方案包含 5 个组成部分，各部分的生成方式和数据来源不同，具体匹配规则如下。

#### 7.2.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 项目基本信息（人工）、方案模板（知识库）、风控矩阵（知识库） |
| 输出 | 审计方案 |

#### 7.2.3 审计方案组成（5 部分）

| 组成部分 | 生成方式 | 内容说明 |
|----------|----------|----------|
| 7.2.3.1 项目基本信息 | 人工输入 | 项目名称、审计目的、被审计单位、审计期间、项目组组长、项目组组员 |
| 7.2.3.2 评价依据 | 调用知识库模板 | 标准内容，直接调用知识库内的方案模板 |
| 7.2.3.3 审计范围 | 自动从风控系统同步 | 根据"标准控制活动选择"底表，获取"业务循环"列和"一级业务流程"列，作为审计范围 |
| 7.2.3.4 审计实施细则 | 模板 + AI 生成 | 见 7.2.4 |
| 7.2.3.5 缺陷认定标准 | 调用知识库模板 | 标准内容，直接调用知识库内的方案模板 |

#### 7.2.4 审计实施细则详解

##### 7.2.4.1 各阶段安排

标准内容，直接调用知识库内的方案模板，包括各阶段的时间规划、里程碑节点。

##### 7.2.4.2 各业务循环审计实施细则

以"销售与收款循环"为例，每个业务循环的审计实施细则包含以下内容：

**（1）审计目标**

验证"销售与收款循环"内部控制的设计合理性和执行有效性，识别重大错报风险和舞弊风险。

> 上方引号内自动匹配"业务循环"名称。

**（2）关键控制点识别**

| 控制活动编号 | 风险场景描述 | 标准控制活动描述 | 检查建议 |
|-------------|-------------|-----------------|---------|
| 取底稿相同字段 | 取底稿相同字段 | 取底稿相同字段 | 智能体生成 |

- AI 角色：专业审计师
- 输入：执行缺陷底稿模板
- 规则：AI 根据底稿模板中每一项风险场景描述内容和标准控制活动描述内容，识别关键控制活动，选择后填入上方表格
- 检查建议：由 AI 根据风险场景描述、控制目标、标准控制活动描述生成

**（3）审计测试程序**

| 控制活动编号 | 测试程序 | 测试方法 | 样本量 | 资料需求 |
|-------------|---------|---------|--------|---------|
| 取底稿相同字段 | 穿行测试 | 智能体生成 | 智能体生成 | 智能体生成 |
| 取底稿相同字段 | 有效性测试 | 智能体生成 | 智能体生成 | 智能体生成 |

- AI 角色：专业审计师
- 输入：执行缺陷底稿模板
- 规则：AI 根据底稿模板中每一项风险场景描述内容、控制目标、标准控制活动描述内容，输出需要执行的程序

**（4）时间安排**

各业务循环审计测试的时间安排，由 AI 根据项目总时间表自动分配。

**（5）人员分工**

各业务循环的审计人员配置，由 AI 根据项目组人员技能背景和工作量自动分配。

### 7.3 生成访谈人员清单及访谈计划

#### 7.3.1 流程描述

根据审计范围和风控矩阵确认待审核的控制活动，由 AI 结合公司部门信息、员工信息，给出各控制活动的访谈对象建议，审计人员审核确认后生成正式访谈计划。

#### 7.3.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 风控矩阵、部门信息、人员信息、审计方案 |
| 输出 | 访谈人员清单及访谈计划 |

#### 7.3.3 访谈对象匹配规则

AI 根据以下维度推荐访谈对象：

- **员工职级**：优先匹配与风险控制活动直接相关的管理层和执行层
- **部门岗位权限**：分析部门岗位权限与业务流程、控制活动的匹配度
- **职能相关性**：优先选择直接负责该控制活动的岗位人员
- **历史参与记录**：参考历史项目中被访谈人员的参与情况

> 待西塞罗系统岗位职责功能上线后，直接对接西塞罗系统获取更精准的岗位职责信息。

#### 7.3.4 访谈计划生成规则

| 约束条件 | 规则说明 |
|----------|----------|
| 时间安排 | 根据审计方案中的时间安排确定 |
| 分组数量 | 根据项目组人员数量和分组情况确定 |
| 访谈对象建议人数 | 每个控制活动至少 1 名访谈对象 |
| 职能划分 | 相同职能的访谈对象优先安排连续时段 |
| 日历排除 | AI 自动排除周末和法定节假日 |
| 输出粒度 | 制定详细到日的访谈计划清单 |

### 7.4 生成访谈问卷

#### 7.4.1 流程描述

根据上一节点确定的访谈人员、风控矩阵、部门信息、人员信息，结合访谈问卷模板和访谈问卷问题库，生成针对每个被访谈人员的个性化访谈问卷。

#### 7.4.2 输入输出

| 项目 | 内容 |
|------|------|

#### 7.4.3 问卷结构

| 模块 | 说明 | 面向 |
|------|------|------|
| 通用部分 | 主要负责业务、负责业务相关的制度、岗位分工情况、流程设计情况、相关系统情况等 | 全部被访谈人 |
| 专业部分 | 根据需要了解的风险场景、控制活动、人员岗位职责等，以问题库为参考生成 | 按岗位定制 |

#### 7.4.4 专业部分生成规则

- **参考依据**：往期访谈的问题清单汇总（访谈问卷问题库）
- **生成策略**：
  - 根据被访谈人岗位职责匹配对应的风险场景和控制活动
  - 根据被访谈人所在业务循环筛选相关领域问题
  - AI 可基于当前控制活动自主生成补充性问题
  - 避免重复性问题，确保每个问题与被访谈人的职能相关

### 7.6 拆分风控矩阵

#### 7.6.1 流程描述

上一节点输出的风控矩阵为汇总表格，但存在以下问题需要在本环节处理：

1. **一对多拆分**：同一控制活动可能有多个制度文件，需要将一行拆分为多行
2. **底稿差异**：设计测试底稿和执行测试底稿的表头不完全相同，需分别输出

#### 7.6.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 风控矩阵（完善后） |
| 输出 | 设计测试底稿、执行测试底稿 |

#### 7.6.3 拆分规则

| 场景 | 处理方式 |
|------|----------|
| 一个控制活动对应多个制度 | 按制度拆分，一行变多行，控制活动编号不变，制度编号/名称按行区分 |
| 多个控制活动对应同一制度 | 保留多行，每行对应一个控制活动 |
| 设计测试底稿字段 | 从风控矩阵提取设计相关的字段（控制活动、风险场景、标准控制活动、制度信息、流程节点） |
| 执行测试底稿字段 | 从风控矩阵提取执行相关的字段（控制活动、风险场景、标准控制活动、测试程序、测试方法） |

### 7.7 评估内控设计缺陷（含打分）

#### 7.7.1 流程描述

根据设计测试底稿（内含根据访谈纪要匹配好的制度编号和名称），按序阅遍相关制度文档，结合历史设计缺陷、设计缺陷类型、流程设置情况，根据评分标准对每个控制活动进行设计合理性评估，并输出设计缺陷类型、缺陷描述和评分。

#### 7.7.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 设计测试底稿、公司制度库、历史设计缺陷库、设计缺陷类型、流程节点设置情况、评分标准 |
| 输出 | 设计测试底稿（完成版） |

#### 7.7.3 设计缺陷类型

| 缺陷类型 | 说明 | 示例 |
|----------|------|------|
| 制度缺失 | 某项风险场景无对应的制度或控制活动覆盖 | 采购验收环节无书面验收制度 |
| 制度不完善 | 制度存在但未覆盖全部关键控制点 | 制度规定了审批流程但未规定审批额度标准 |
| 制度设计不合理 | 制度设计无法有效降低风险至可接受水平 | 审批权限设置过于集中，未体现不相容职务分离 |
| 制度未更新 | 制度未根据业务变化或法规变更及时修订 | 制度仍引用已废止的法规条款 |
| 流程缺失 | 关键业务环节缺失必要的流程设计 | 无退货处理流程 |
| 流程与制度不匹配 | 实际流程与制度描述不一致 | 制度要求三级审批，流程仅设置两级 |
| 控制目标不明确 | 控制活动缺乏明确的可衡量控制目标 | 控制活动描述模糊，无法判断控制是否有效 |

#### 7.7.4 设计缺陷评分标准

| 评级 | 分值区间 | 判定标准 |
|------|----------|----------|
| 无缺陷 | 5 分 | 制度设计完整，覆盖全部关键风险点，流程与制度一致，控制目标清晰可衡量 |
| 一般缺陷 | 3 分 | 存在次要的设计缺失，但不影响主要控制目标的实现，可通过小幅修订完善 |
| 重要缺陷 | 1 分 | 存在重要的设计缺失，可能导致某项控制目标无法实现，需要较大幅度修订 |
| 重大缺陷 | 0 分 | 存在根本性的设计问题，可能导致多个控制目标失效，需要重新设计控制体系 |

#### 7.7.5 评估规则

| 评估维度 | 规则说明 |
|----------|----------|
| 制度完整性 | 检查是否有制度覆盖该控制活动，制度内容是否完整 |
| 制度充分性 | 检查制度设计是否足以将风险降低到可接受水平 |
| 制度时效性 | 检查制度是否为最新版本，是否与当前业务匹配 |
| 流程匹配度 | 检查实际流程与制度规定是否一致 |
| 控制目标匹配 | 检查控制活动的设计是否指向明确可衡量的控制目标 |
| 历史缺陷参考 | 参考历史设计缺陷中相同或相似制度的过往问题 |
| 流程节点设置 | 参考手工上传的各流程节点设置情况，辅助制度与流程匹配 |

#### 7.7.6 流程节点设置说明

流程节点设置信息由用户手工上传各流程的节点设置情况，用于制度、控制点和流程的匹配。后续可考虑做成知识库或系统对接的方式自动获取。

### 7.9 评估内控执行缺陷（含打分）

#### 7.9.1 流程描述

根据系统对接的数据或人工上传的测试资料，结合流程测试数据、历史执行缺陷、执行缺陷类型和评分标准，判断控制活动的执行有效性，输出执行缺陷类型、缺陷描述和评分。

#### 7.9.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 业务数据 / 测试资料、历史执行缺陷库、执行缺陷类型、评分标准、流程测试数据 |
| 输出 | 执行测试底稿（完成版） |

#### 7.9.3 执行缺陷类型

| 缺陷类型 | 说明 | 示例 |
|----------|------|------|
| 未执行 | 设计的控制活动在实际操作中未被执行 | 制度要求双人复核，实际只有一人签字 |
| 执行偏差 | 控制活动被执行但方式与设计要求不符 | 审批流程与制度规定的审批路径不一致 |
| 执行频率不足 | 控制活动的执行频率低于设计要求 | 制度要求每日盘点，实际每周盘点一次 |
| 执行记录缺失 | 控制活动已执行但缺乏完整的执行记录 | 审批单缺失，无法证明控制已执行 |
| 职责分离不当 | 不相容职务未有效分离 | 同一人同时负责采购申请和采购验收 |
| 越权操作 | 操作超越授权范围 | 下级人员代行上级审批权限 |
| 系统控制失效 | 系统自动控制未按设计运行 | 系统未拦截超出预算的采购订单 |
| 样本偏差 | 控制执行存在选择性偏差 | 仅对部分业务执行了规定的控制 |

#### 7.9.4 执行缺陷评分标准

| 评级 | 分值区间 | 判定标准 |
|------|----------|----------|
| 无缺陷 | 5 分 | 控制活动按设计被有效执行，执行记录完整，无执行偏差 |
| 一般缺陷 | 3 分 | 存在次要的执行偏差或偶发性的执行缺失，未影响控制目标的实现 |
| 重要缺陷 | 1 分 | 存在系统性的执行偏差或执行缺失，可能导致控制目标部分失效 |
| 重大缺陷 | 0 分 | 存在普遍性的执行缺失或严重执行偏差，导致控制目标整体失效 |

#### 7.9.5 评估规则

| 评估维度 | 规则说明 |
|----------|----------|
| 穿行测试 | 从业务发起到结束全程追踪，验证控制活动是否按设计执行 |
| 有效性测试 | 抽取足够样本量，统计控制执行的符合率 |
| 历史缺陷参考 | 参考历史执行缺陷中相同或相似控制活动的过往问题 |
| 业务数据分析 | 结合业务系统数据进行自动化比对和异常识别 |
| 流程测试数据 | 参考流程测试中的运行数据和异常报告 |

### 7.10 内控缺陷输出及确认

#### 7.10.1 流程描述

将 7.7 和 7.9 输出的设计缺陷、执行缺陷清单与业务部门进行沟通确认，经多轮确认后输出最终版缺陷清单。包含三个步骤：业务初步确认、输出缺陷汇总及沟通确认表、与业务负责人确认。

#### 7.10.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 设计测试底稿（完成版）、执行测试底稿（完成版） |
| 输出 | 最终版缺陷清单、缺陷汇总及沟通确认表 |

#### 7.10.3 步骤一：业务初步确认（7.10.1）

将 7.7 和 7.8（即 7.7 评估内控设计缺陷和 7.9 评估内控执行缺陷）输出的设计缺陷清单和执行缺陷清单明细，与业务部门进行初步沟通确认（通过发布任务或邮件方式）。

确认内容：

| 字段 | 说明 |
|------|------|
| 所属循环 | 缺陷所属业务循环 |
| 缺陷类型 | 设计缺陷 / 执行缺陷 |
| 缺陷认定 | 一般缺陷 / 重要缺陷 / 重大缺陷 |
| 缺陷简述 | 缺陷的简要概括 |
| 缺陷描述 | 缺陷的详细说明 |
| 整改建议 | 初步的整改建议 |
| 整改责任部门 | 负责整改的部门 |
| 业务反馈 | 业务部门对缺陷的初步反馈意见 |

#### 7.10.4 步骤二：输出缺陷汇总及沟通确认表（7.10.2）

根据步骤一输出的缺陷明细，由 AI 按照以下维度进行汇总统计：

| 事业部 | 一阶管理部门 | 业务循环 | 设计缺陷数量 | 执行缺陷数量 | 一般缺陷数量 | 重要缺陷数量 |
|--------|-------------|----------|-------------|-------------|-------------|-------------|
| 添可 | 财经管理中心 | 销售与收款 | 5 | 18 | 5 | 2 |

维度说明：

| 汇总维度 | 说明 |
|----------|------|
| 事业部 | 按事业部分组汇总 |
| 一阶管理部门 | 按一级管理部门分组汇总 |
| 业务循环 | 按 19 个业务循环分别汇总 |
| 缺陷类型 | 区分设计缺陷和执行缺陷 |
| 缺陷认定 | 区分一般缺陷、重要缺陷、重大缺陷 |

#### 7.10.5 步骤三：与业务负责人确认（7.10.3）

根据 7.10.4 输出的缺陷明细表和沟通确认表，与各业务负责人进行面对面或线上沟通确认：

- 逐项确认缺陷事实的准确性
- 确认缺陷认定的合理性
- 收集业务部门对整改措施的初步意见
- 确认整改责任部门和整改时限

### 7.11 内控评价总体打分

#### 7.11.1 流程描述

根据 7.10 确认后的缺陷清单，对各控制事项进行打分，并按事业部、循环维度计算总分，输出得分明细和得分汇总表。

#### 7.11.2 输入输出

| 项目 | 内容 |
|------|------|
| 输入 | 经业务确认的缺陷清单 |
| 输出 | 缺陷清单（增加"得分"列）、得分汇总表 |

#### 7.11.3 输出内容

**输出 1：缺陷清单 + 得分**

在 7.10 输出的缺陷清单基础上，增加以下字段：

| 新增字段 | 说明 |
|----------|------|
| 设计缺陷评分 | 对应控制活动的设计缺陷评分（0-5 分） |
| 执行缺陷评分 | 对应控制活动的执行缺陷评分（0-5 分） |
| 综合评分 | 设计缺陷评分和执行缺陷评分的综合得分 |

**输出 2：得分汇总表**
